# 5 智能体推理与规划

### 本章涵盖

* LLM 如何推理和规划
* 指导智能体进行推理和规划
* 智能体的高级规划
* 使用顺序思考 MCP 服务器

对 LLM 而言，推理涉及两种不同的认知操作。**分解**是将问题拆解成更小的子问题（这项任务的组成部分是什么）。**规划**是为解决这些子问题选择顺序、依赖关系和方法（以什么顺序、使用什么工具、各部分如何连接）。

这两种操作都是必要的，而且它们可以独立失败。一个智能体可以正确分解任务但规划得很差，或者在一个糟糕的分解基础上进行正确规划，而在这两种情况下，产生的失败表现是不同的。指出这种区别之所以重要，是因为修复方法取决于哪个操作出了问题。

没有这两者，智能体利用自主性做出决策、执行行动和实现目标的能力就会受到限制。正如我们将看到的，这两种操作的质量往往始于为智能体提供动力的 LLM 以及我们围绕它构建的推理结构。

## 5.1 理解 LLM 推理与规划

分解和规划最容易通过一个程式化的例子来理解。为了实现“离开家”，一个简单的计划可能是：穿好衣服、检查烤面包机、拿钥匙、离开。

对于日常任务，真正的人类认知并非以这种方式工作。我们依赖习得的常规和从具身经验中构建的丰富世界模型，而这两者都是 LLM 所缺乏的。这就是为什么智能体需要显式的分解和规划结构来处理人类认知隐含完成的工作。

非推理型 LLM 在一次前向传递中产生输出，没有显式的思考阶段。如果没有如思维链（CoT）提示这样的推理框架，这些模型往往会短路成听起来合理但肤浅的答案，尤其是在多步骤问题上。这种限制在于多步骤推理的质量，而非模型会尝试的任务数量。

推理型模型使思考过程显式化。它们在产生最终输出之前，将计算资源分配给一个可见的思考阶段，这让它们能够以额外的令牌和延迟为代价，解决更困难的问题。区别不在于是否发生推理，而在于推理是隐式的、受单次传递限制，还是显式的、并有扩展空间。

思维链提示介于这两者之间。它鼓励非推理型模型在其输出中生成中间步骤，这提高了多步骤任务的性能，但并未给模型带来它原本不具备的新能力。CoT 对强模型的帮助大于对弱模型的弥补，在假设某种提示技术能够改变系统的底层推理能力之前，了解这一点是值得的。

幸运的是，像思维链这样的提示策略已经出现，它们使推理步骤在模型的输出中变得显式。如前一节所述，这与其说是赋予模型新能力，不如说是将其从短路到肤浅答案，转向生成它可以依赖的中间步骤。

另一个发展是专用推理模型的兴起。OpenAI 的“o”系列是最清晰的例子，它通过显式的测试时计算和强化学习训练，在回答之前实现扩展的内部思考。其他前沿模型，包括 Claude 和 Gemini，现在也暴露了一个可配置的推理模式，支持类似的思考阶段，尽管各提供商之间的底层训练和架构有所不同。

这种区别在实践中很重要。具有显式推理基础设施的模型（OpenAI 的 o 系列、启用了扩展思考的 Claude、具有推理模式的 Gemini）被设计为在思考阶段投入大量计算资源，从而导致相应的延迟和令牌成本。没有该模式的通用模型依赖于像 CoT 这样的提示技术来显示中间步骤，这更便宜、更快，但受限于模型的单次传递能力。图 5.1 比较了一个没有推理基础设施的 LLM（上图）和一个针对复杂问题具有扩展推理能力的 LLM（下图）。

![](./images/CH05_F01_Lanham4.png)

##### 图 5.1 针对推理型和非推理型模型的不同提示策略

如图所示，非推理型模型需要一个明确的目标陈述，并辅以推理和规划策略，以产生结构化的中间步骤。推理型模型在没有这种框架的情况下产生结构化的中间步骤，它们依赖于训练期间学到的推理时技术。这种行为仍然是概率性的，并且对提示敏感，因此同一个模型可能在一个请求上仔细推理，而在另一个请求上短路。

这使得评估比“它是否推理”的二元判断更加细致。实际问题是：模型在你关心的任务上推理的可靠性如何，这种推理在令牌和延迟方面花费多少成本，以及相对于带有 CoT 提示的非推理型模型所获得的收益是否值得这种差异。这些是驱动实际智能体工作中模型选择的问题，而非哲学问题。

推理型模型越来越成为标准，但模型的选择并不消除设计智能体推理结构的需要。从业者工具包中有两种主导模式：思维链（CoT）和推理-行动-观察（ReAct）。两者都值得理解，因为它们解决不同的问题，并且与推理型和非推理型模型都能很好地结合。

CoT 将推理结构化为单一的线性传递。模型在给出最终答案之前，在其输出中生成中间步骤。当问题可以通过分阶段推理而无需外部信息来解决时，这种方法效果很好。现代推理型模型本身就是在 CoT 轨迹上训练的，因此在推理型模型上进行 CoT 提示更接近于强化一种已学习的模式，而不是引入一种新模式。

ReAct 引入了一个反馈循环。智能体交错进行推理步骤、工具行动和结果观察，使其能够将新信息纳入后续推理中。ReAct 是大多数生产型智能体和大多数多智能体系统的结构基础，因为它是将内部思考与外部行动连接起来的模式。

关于这些模式做什么和不做什么的说明：两者都塑造和影响模型的推理方式；两者都不在严格意义上控制推理。即使有结构化的提示，LLM 仍然是概率系统，因此具有相同 CoT 或 ReAct 设置的同一智能体在不同运行中可能产生不同的推理轨迹。

新旧模型之间的能力差异也比“新模型有显式推理基础设施”更广泛。规模、训练计算量、数据质量和训练后策略都有贡献，其中任何一个单独都不能解释这种差距。将推理模式视为众多杠杆中的一个，而不是能弥补其他一切的技巧。

### 5.1.1 思维链推理

思维链（CoT）提示引导 LLM 逐步推理问题，而不是直接跳到答案。在 CoT 方法中，模型在产生最终答案之前生成一系列中间推理步骤（一条“思维链”）。这个过程有效地让 LLM 通过分解来解决复杂任务，显著提高了智能体常见的复杂目标的性能。

在 CoT 提示中，我们向 LLM 提供它应如何推理的示例。我们可以通过打开 ChatGPT，选择非推理型或推理型模型，并输入如下示例提示来轻松演示这一点。

##### 清单 5.1 01_demonstrating_CoT_reasoning.md

```
# 基础时间旅行问题
在一部科幻电影中，亚历克斯是一个时间旅行者，他决定回到过去，目睹一个发生在100年前的著名历史事件，
该事件持续了10天。他在事件开始前三天到达。
然而，在过去度过了六天后，他向前跳跃了50年，并在那里停留了20天。
然后，他穿越回去目睹了事件的结束。
亚历克斯在目睹事件结束之前，在过去度过了多少天？
```

这个问题存在一些细微差别，使得没有推理能力的模型难以处理，有时甚至对有推理能力的模型也具有挑战性。正确答案是 26 天，但根据你选择的模型，你可能会得到 6、13 或 27 天。有些模型甚至可能一次给出一个答案，下一次给出不同的答案。

##### 正确答案

要得到 26 天的答案，我们可以将总天数计算为各次跳跃时间的总和，但为了使计算更具体，让我们假设一个起始日期为 2026 年：

* 跳跃 1：在事件开始前 100 年零 3 天（1926 年），在 1926 年度过 6 天
* 跳跃 2：向前 50 年到 1976 年，仍在亚历克斯的过去；在 1976 年度过 20 天
* 跳跃 3：向后 50 年到 1926 年事件的最后一天，目睹事件，并假设在过去没有花费额外天数
* 跳跃 4：向前到 2026 年，现在

我们将 1926 年度的 6 天与 1976 年度的 20 天相加，得到在过去度过的总天数为 26 天。

LLM 在这个问题上的困难在于它无法改变视角和参照系。我们可以使用 CoT 提示策略，通过在提示中定义思考过程来帮助模型进行推理。以下清单显示了添加的 CoT 推理，我们指示使用它来解决这类问题。

##### 清单 5.1 01_demonstrating_CoT_reasoning.md（续）

```
#带推理 - 附加到提示
CoT 推理策略
步骤 1：将当前年份定义为第 0 年，将过去定义为负数年份。
步骤 2：绘制旅行者的旅程图：去了哪里，停留了多久。
步骤 3：确定哪些时间段算作“过去”（从旅行者的视角）。
步骤 4：追踪在过去的每个时间段内度过的天数。
步骤 5：将事件结束前在过去度过的所有天数相加。
步骤 6：验证计算是否包含了所有过去的时间，并排除了现在/未来的时间。
```

将可复用的 CoT 思考过程和步骤添加到提示中，为 LLM 提供了解决问题的步骤。当你输入完整的提示（先输入问题，再输入 CoT 策略）时，LLM 会针对每个步骤输出结果。在大多数情况下，添加 CoT 将有助于提供正确答案，但某些模型可能仍然会遇到困难。

当模型在这方面遇到困难时，不要认为 CoT 有问题；相反，步骤可能需要调整以适应你特定模型的语法和风格。图 5.2 比较了 LLM 在没有提示推理和使用 CoT 推理的情况下回答问题的过程。

![](./images/CH05_F02_Lanham4.png)

##### 图 5.2 使用 CoT 提示（左图）和不使用任何显式推理指令（右图）时 LLM 思考过程的比较

如图所示，使用 CoT 提示时，LLM 将输出过程中的每一步思考。这在确定思考过程可能在哪里出错时非常有帮助。还应注意，CoT 不是一个神奇的过程；它通常需要相当长的时间才能让 LLM 按照你想要的方式思考。

使用 CoT 提示策略的一个好处是能够检查 LLM 的思考过程，确保 LLM 按预期进行推理和规划，并构建该思考过程。使用这种模式的缺点是额外的令牌使用、更高的延迟和更多的输出。尽管如此，CoT 在控制最终输出方面的能力是有限的，而且正如我们将在下一节中看到的，有更好的方法来管理这一点。

### 5.1.2 推理、行动、观察：ReAct 范式

虽然 CoT 将推理完全保持在内部，但 ReAct 范式在交互循环中交替进行推理和采取行动。模型交错进行思考步骤（关于做什么的自然语言推理）和行动步骤（使用外部工具或查询 API 的命令）。每次行动后，模型都会观察结果，根据该反馈更新其推理，并继续循环。

反馈循环是使 ReAct 与 CoT 有实质性区别的原因，而不仅仅是在于工具调用。在某个地方插入工具调用的单次 CoT 传递仍然是单次传递。ReAct 是一个闭环，其中每次观察都输入到下一个推理步骤中，这使智能体能够从错误行动中恢复、纳入新信息并在任务进行中调整其计划。

ReAct 的思考、行动和观察节奏让智能体在每一步都能收集新信息，并调整下一步的操作。ReAct 是反应式的，而不是计划驱动的：模型一次推理一步，并根据最新的观察决定下一步行动，而不是预先制定一个全局计划。驱动循环（调用模型、执行行动、返回观察结果、再次调用模型）位于智能体框架中，而不是模型本身内部。

图 5.3 将 ReAct 模式显示为推理、行动和观察阶段的序列图。该图使外部编排可见，这值得关注，因为它是决定智能体在实践中行为方式的部分。

![](./images/CH05_F03_Lanham4.png)

##### 图 5.3 序列图中的 ReAct 推理范式。智能体思考过程发生在循环中，LLM 首先推理、规划，然后执行计划。每次执行任务（工具）后，LLM 观察输出，然后推理是需要继续循环还是计划已完成。

在图中，查询被发送到智能体/助手，它启动一个处理循环。首先，助手推理回答查询所需的步骤。在构建了这个思考/步骤计划之后，助手执行一个工具/MCP 调用并观察结果。如果助手确定目标已完成，循环停止并输出最终回复。如果没有，助手会推理当前的步骤计划，然后继续执行计划或创建一个新的思考/步骤计划并继续循环。

这是许多推理型 LLM 用来解决任务的相同过程。如果模型提供此选项，你可以通过观察其思考、行动和观察过程来观察到它。我们可以通过引入规划来协调 LLM 的思考，从而将这一点扩展到 CoT 和 ReAct 之外。

### 5.1.3 LLM 规划

除了单个问题或工具调用之外，LLM 还可以进行更高层次的战略规划，以应对复杂的多步骤目标。规划意味着模型在思考/步骤执行之前或期间，制定一个战略大纲或行动计划。LLM/智能体在更长的时间范围内进行规划，并考虑多个 CoT/ReAct 执行循环。

规划模式与 CoT 和 ReAct 的不同之处在于智能体对问题的看法存储在哪里。CoT 在单次前向传递中通过线性推理链处理一个查询。ReAct 一次一步地决定行动，每一步的决策都基于最新的观察结果，而不是更长期的视角。

规划模式会产生一个显式的计划——存储在模型之外的内存、暂存器或对话上下文中——智能体在每一步读取该计划，以协调朝着总体目标的工作。LLM 本身在调用之间没有持久状态，也没有内部世界模型，因此全局视图是架构的属性，而不是模型的属性。这种区别在设计规划智能体时很重要，因为计划跟踪的所有内容（子任务、依赖关系、中间结果、进度）都必须被显式化，并存储在智能体可以重新读取的地方。

给定查询或目标，LLM 非常擅长制定计划。它们可以使用工具执行该计划并观察结果，这与我们刚刚观察到的 ReAct 范式并无不同。以下清单演示了我们可以用来要求 ChatGPT/Claude/Gemini 为期望目标制定计划的提示。

##### 清单 5.3 01_planning_san_fran_trip.md

```

第1天：探索OpenAI总部和Exploratorium的AI展览
```

在任何基础模型中输入查询。你很可能会在后台看到一些活动，例如网络搜索或其他帮助模型产生更好输出的查询。模型并没有在思考如何规划；它是在生成构成计划的令牌，利用训练期间学到的模式。

例如，我们从旧金山的这个行程（计划）开始，但发现Exploratorium在第一天关闭，并且没有初创公司可以参观。根据这些新信息，我们将更新后的行程输入一个基础LLM，并要求其修改计划。以下演示了这一过程以及LLM可能回复的输出。

如果我们将我们自己替换为智能体，我们可以看到智能体如何应用这些相同的原则，如下一节所示。

## 5.2 指导智能体进行推理和规划

理解推理和规划如何工作的下一步是将示例集成到智能体中。智能体是围绕LLM构建的系统，将模型与工具、记忆、编排逻辑以及与环境交互的能力相结合。LLM是推理组件。其他所有组件都将这种推理转化为影响世界的工作。

### 5.2.1 将CoT应用于智能体

我们将从通过CoT赋予智能体推理能力的第一步开始。清单5.5使用一个简单的智能体实现了CoT。在这个例子中，我们赋予智能体一个角色，并给出逐步思考问题的简单指令。

##### 清单5.5 02_CoT_agent.py

```python
cot_agent = Agent(
   name="TimeTravelerCoT",
   instructions=(    #1
       "你是一个时间旅行问题解决者。"
       "逐步计算出解决方案，然后给出最终答案。"
   ),
)

question = (    #2
   "从2025年开始，你穿越到10年前，然后穿越到5年后。"
   "你最终到达哪一年？"
)

result = asyncio.run(   #3
   Runner.run(    #4
       cot_agent, 
       input=question))
print(result.final_output)

OUTPUT
让我们逐步解决这个问题：
1. **起始年份**：2025
2. **穿越到10年前**：
  - 2025 - 10 = 2015
3. **穿越到5年后**：
  - 2015 + 5 = 2020

 #1 设置带有指令的CoT智能体  
#2 我们将要求智能体解决的问题  
#3 异步运行
```

## 5.3 智能体的高级推理模式

并非所有智能体都需要显式的推理结构。执行范围明确、单步任务的简单智能体（查找一个值、调用一个工具、生成固定格式的响应）通常不会从CoT或ReAct中受益，因为没有多步决策需要结构化。当任务需要多个决策、依赖于中间结果，或者复杂性足以使单次前向传递产生不可靠输出时，推理模式才值得其成本。

当你确实需要推理时，CoT或ReAct将覆盖大多数情况。当你构建处理更困难目标的更复杂智能体系统时，像思维树（ToT）和Reflexion这样更专业的模式就值得使用了。

### 5.3.1 思维树

思维树（ToT）是对候选推理路径的刻意搜索过程，而不是与CoT相同意义上的推理框架。CoT产生一条单一线性链的中间步骤，而ToT探索多个分支，评估它们，修剪没有前途的分支，并在推理路径失败时回溯。该方法在专用推理模型出现之前就已引入，解决的是单次传递模型在需要前瞻和探索的问题上的特定弱点。

ToT对于战略探索比任何单一路径的深度更重要的场景很有帮助。规划任务、谜题和需要多步移动的游戏是典型的适用场景。现代推理模型通过其扩展的思考阶段原生地覆盖了其中的一些领域，这缩小了你需要专门使用ToT的场景，尽管在具有明确分支和评估标准的问题上，显式的搜索结构仍然胜出。

一个值得关注的实践问题：ToT不能像CoT那样通过单个提示完全调用。分支和评估部分可以通过提示模型生成几个候选思维然后评分来近似，但定义ToT的修剪和回溯由外部编排代码处理。LangGraph和自定义实现是在生产环境中运行ToT的典型方式。

图5.4演示了这种模式的第一步实现。图中所示的ToT模式有很多内容。将这个过程看作一个步骤列表可能会有帮助，这些步骤分解了这种复杂的推理策略：

![](./images/CH05_F04_Lanham4.png)

##### 图5.4 思维树过程的部分工作流，其中思维分支成节点，每个节点被执行以确定要遵循的最佳路径

1. 智能体针对给定问题生成几个初始想法或可能的解决步骤，形成思维树的第一层。
2. 对于每个候选想法，智能体通过思考下一步来扩展它，从而并行地生长多个推理分支。
3. 在扩展到一定深度或步数后，智能体（或启发式方法）评估每个分支叶节点的部分解决方案。这种评估可能估计该分支导致正确解决方案的可能性。例如，可以提示模型将每个分支评分为“有希望”或“不太可能”达到目标。
4. 基于这些评估，算法修剪低潜力的分支，并选择最有希望的分支继续扩展。然后该过程迭代：扩展之后是评估和修剪。
5. 当在其中一个分支上找到满意的解决方案，或达到最大搜索深度时，搜索终止，在后一种情况下，选择探索过的思维中得分最高的解决方案作为最终答案。

我们可以将整个过程改编成智能体系统，如清单5.7所示。但该清单仅评估初始思维步骤作为一个分支。对于完整的解决方案，我们必须对每个思维步骤迭代相同的模式。

### 5.3.2 Reflexion

Reflexion 是一种自我批评模式，其中智能体评估自己的输出，生成关于哪里出错的反馈，并尝试再次解决，将该反馈纳入其上下文中（图 5.5）。批评可能是一个提示、一个错误分析或对下一次尝试的具体建议。多次尝试之间的改进来自于模型基于更丰富上下文的条件化，而不是任何实际的学习，因为没有模型权重被更新。

![](./images/CH05_F05_Lanham4.png)

##### 图 5.5 逐步流程图中的 Reflexion 推理策略。流程中的每一步都展示了可能发生在 LLM 或确定性代码中的任务、决策或输出。

早期 Reflexion 论文和教程中常见的“学习”框架具有误导性。没有训练步骤，没有策略优化，并且除非我们显式保存它，否则智能体没有过去会话的记忆。看起来像学习的东西，实际上是当输入包含对其先前输出的批评时，模型产生更好的输出。

Reflexion 通常比 ToT 更便宜，因为它一次探索一条路径并进行迭代细化，而不是并行维护多个分支。它非常适合困难或模糊的问题，其中第一次尝试很可能是错误的，但失败信息足够丰富，使得批评可以指导更好的第二次尝试。与 ToT 的权衡是深度与广度：Reflexion 通过修订在一条路径上走得更深，而 ToT 在多条路径上更广泛地探索。

图中的过程演示了 Reflexion 的一个抽象实现。在实践中，几个步骤将由一个生成智能体和一个批评智能体实现，并有一些确定性代码来检查硬限制，如重试次数甚至最终输出。

正如可能显而易见的那样，这些推理策略通常建立在前面的策略之上。这意味着它们依赖于早期策略的实现，例如使用 CoT 产生初始计划，如图中的流程图和下一个清单中的代码示例所示。该代码将用于尝试解决清单 5.1 中考虑的那个困难的时间旅行问题。

理想情况下，使用 Reflexion 策略进行推理、规划和通过反馈进行评估，可以使基础求解器智能体随着时间的推移而改进。这种模式的一个显著缺点是它需要知道正确的解决方案才能提供反馈。这并不理想，但如果你有大量类似问题的集合，你可以使用 Reflexion 解决几个问题，然后将反馈提示填充到记忆/存储中，以解决集合中的所有问题。

### 5.3.3 为你的智能体选择正确的模式

推理策略很强大，但并非总是适用于每个用例。了解何时何地使用每种模式来自于构建推理智能体的经验。尽管如此，对于日常用例，有一些既定的最佳实践，如表 5.1 所示。

##### 表 5.1 智能体推理策略常见用例的最佳实践

| 策略 | 最适合 | 计算成本 | 何时不使用 | 具体示例 |
| :--- | :--- | :--- | :--- | :--- |
| **思维链** | 逻辑谜题、数学问题 | 低 | 简单的事实查询 | 类似数独的谜题 |
| **ReAct** | 知识密集型任务 | 中 | 不需要外部工具时 | 检索增强问答 |
| **思维树** | 复杂规划、游戏 | 高 | 时间敏感的应用 | 策略游戏搜索 |
| **Reflexion** | 迭代改进任务 | 中高 | 一次性问题 | 带迭代提示的代码调试 |

如你所见，这四种推理策略将覆盖大多数问题。但它们都依赖于简单的 LLM、CoT 或 ReAct 进行规划，这最终可能限制整体实现。幸运的是，Anthropic 提供了一个可通过 MCP 使用的结构化顺序思考服务器的参考实现。

## 5.4 使用顺序思考 MCP 服务器

尽管顺序思考（ST）MCP 服务器有多种实现，但我们将专注于 Anthropic 的参考实现。你的智能体可以使用这个服务器作为推理和规划的工作记忆暂存器，存储中间输出，以便后续步骤可以读取并在此基础上构建。_顺序思考_这个名称有点误导，因为服务器本身并不思考；它为智能体思考的输出提供外部存储，智能体在每一步重新读取这些输出，以保持跨调用的连续性。

允许你的智能体存储其思维，使其能够更具战略性地思考。当它处理复杂问题时，它可以创建高层战略计划并将其保存在 ST 服务器中。然后，当它处理每个任务和目标时，它可以根据需要审查和修订全局战略计划。

重要的是要记住，工具描述是智能体看到的，并指导它如何解释工具的使用。请注意，这个工具（`SequentialThinking`）本身并不思考或推理。它只是一个暂存器和思维跟踪器。其他实现的顺序思考服务器使用 LLM 和智能体来推理和规划思维。

你可能还会注意到，ST 允许智能体使用 ReAct 和 ToT 规划、推理和执行策略。要在你的提示中使用这些策略，你只需要适当的指令。

### 5.4.3 使用顺序思考的高级推理

顺序思考服务器允许你组合不同的推理模式，并帮助你跟踪每种模式如何与智能体协同工作。对于我们的最后一个推理示例，我们将结合本章中使用的所有推理模式：CoT 用于规划，ReAct 和 ToT 用于计划执行和评估，以及 Reflexion 用于反馈，以便智能体推导出正确的计划来解决问题。清单 5.12 展示了这些推理策略如何组合。同样，组合多种推理模式并使用像 ST 服务器这样的工具最适合复杂、模糊的任务，而不是需要低延迟和快速响应的任务。

当你运行此代码示例时，预计它会花费一些时间并消耗相当多的令牌。这是因为我们添加了 ToT 和 Reflexion 推理策略，它们计算量大且消耗令牌多。此智能体的结果将根据你用来驱动 LLM 的底层模型而有很大差异。更高级的推理模型将表现良好，但其他模型可能不行。

## 总结

* 大语言模型默认情况下并不“思考”；它们是令牌预测器，因此智能体必须注入结构化的推理和规划来实现多步骤目标。
* 思维链 (CoT) 提示将模型的隐藏直觉转变为显式的、逐步的思考，易于调试，但代价是额外的令牌和延迟。
* ReAct 通过工具调用增强了 CoT：推理 > 行动 > 观察 > 重复，让智能体在迭代优化其计划时动态收集信息。
* 高层规划超越了单链：智能体可以要求 LLM 起草一个战略大纲，然后根据现实世界的反馈进行修订。
* 思维树 (ToT) 并行探索许多分支，修剪失败者并扩展有希望的路径。它对于复杂的搜索任务非常强大，但极其消耗令牌。
* Reflexion 将求解器-批评者循环包裹在任何推理策略周围：批评者提供反馈，求解者修订，循环重复，直到答案通过自定义检查。
* 选择策略取决于任务：CoT 用于逻辑谜题，ReAct 用于工具密集型查找，ToT 用于深度规划，Reflexion 用于迭代改进；随着需求变化混合搭配。
* 顺序思考 MCP 服务器充当通用的“暂存器”工具；智能体在那里写入、修订和分支思维，同时仍然在外部使用 ReAct 或 ToT 模式。
* 组合策略可以扩展推理能力：例如，CoT 起草计划，ReAct 加 ToT 执行或分支它，Reflexion 自我评分并重试；预期高延迟但高可靠性。
* 护栏、模式和类型化输出仍然至关重要；推理输出应像任何其他智能体 I/O 一样进行验证，以避免级联错误。
* 模型选择很重要：较新的原生推理模型（例如，GPT-4o 系列）处理这些模式所需的提示更少，但即使是基础模型，在得到适当指导时也能进行推理。
* 保持工具列表精简；每个额外的工具都会增加 ReAct 循环和顺序思考调用的负担，因此将每个智能体的工具范围限制在 < 10 个高度相关的工具。
* 一个生产就绪的推理智能体结合了结构化提示、正确的推理策略、用于思维跟踪的顺序思考以及用于验证的护栏，产生能够自主调整、重试和成功的计划。
