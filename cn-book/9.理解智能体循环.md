# 9 理解智能体循环

### 本章涵盖

* 揭示智能体循环的三个层次
* 深度研究智能体的循环
* 多智能体编排循环
* 构建协作式智能体循环

早在第1章，我们就探讨了核心的智能体循环模式，即我们称之为SPAL（感知-规划-行动-学习）的过程循环。在本章中，我们将超越内部智能体循环，研究扩展到智能体外部的循环模式。这些模式用于制定长周期、目标驱动的智能体，例如那些涉及深度研究的智能体，或者更广泛地，在所谓的“智能体协作”工作流中。

使用智能体循环来赋能智能体，在单智能体和多智能体系统中引入了新的指挥与控制层级。本章探讨了智能体循环的三个层次。这些层次从我们已经知道的内部循环（SPAL），到长周期目标执行，再到多智能体系统中的元引导循环。在此过程中，我们将研究每个层次的实际应用，以及如何将这些模式引入您的智能体工作流中。

## 9.1 揭示智能体循环的三个层次

内部循环（SPAL）是智能体的第一个或核心层。这个循环允许智能体迭代、发现、推理和执行任务以完成目标。但还有两个智能体循环层可以扩展智能体的工作：第2层，任务循环；第3层，元循环。

### 9.1.1 第1层：内部循环（感知-规划-行动-学习）

第1层智能体循环通过将智能体的执行包装在一个受控的循环中工作，该循环重复直到智能体的目标满足或触发终止条件。图9.1说明了第1层核心智能体循环架构。如您所见，智能体从一个目标开始，并感知当前的内部状态（包括来自先前迭代的任何累积上下文）。然后它规划下一步行动，使用其工具执行该行动，评估结果，并更新内部状态。然后它决定目标是否已达成，如果没有，则循环回来，携带它学到的所有内容。

![](./images/CH09_F01_Lanham4.png)

##### 图9.1 核心智能体循环。智能体循环通过感知、规划、行动和评估步骤，直到目标满足或触发终止条件。内部状态在迭代间累积，使每个循环比上一个拥有更多上下文。

使这个循环成为_智能体循环_，而不仅仅是代码中的_while_循环，是因为智能体本身在每一步做出决策。智能体决定如何完成目标，调用哪个工具，结果是否足够好，以及何时停止。开发者定义目标、提供工具并建立边界，但智能体驱动迭代。

从图中，注意一些重要的术语已被标识出来。_目标_，智能体将承担什么；_计划_，智能体将如何执行；_工具_，智能体将如何行动；_状态_（内部），智能体如何保持上下文（记忆）；以及_感知/决策_，智能体如何确定何时完成。在第1层，所有这些元素都在智能体内部。

每个智能体层级循环都依赖于这些核心元素：目标、计划（迭代策略）、状态和决策（终止条件）。正确处理好这些是区分一个能收敛到优秀输出的智能体与一个永远运行或过早放弃的智能体的关键。

### 9.1.2 第2层：任务循环

智能体循环的第2层是关于将核心元素（目标-计划-状态-决策）外部化到智能体之外。这扩展了智能体可以努力实现的目标范围，从简单目标到非常复杂的目标。图9.2展示了我们可能要求智能体执行的短期和长期目标的例子。它还扩展了第1层智能体循环的概念到第2层智能体循环。

![](./images/CH09_F02_Lanham4.png)

##### 图9.2 短期与长期目标的比较。您可以看到这个过程如何要求将单智能体内部循环提升到第2层基于外部任务的智能体循环。

我们转移到二级外部循环，为长期计划和扩展状态持久性腾出空间。在内部，智能体仍然维护计划和内部状态，但这些部分在智能体的内部周期中是短暂的。为了实现长期目标，智能体系统需要将计划和状态持久化到外部。

通过外部化智能体的计划和状态，单个智能体可以跨多个短期目标工作，以实现深度研究等长期目标。我们这样做不仅是因为我们不想压垮LLM的上下文窗口。我们还因为通用的LLM在设计上不适合长时间运行的迭代。最近，面向智能体的LLM开始流行，但它们仍然面临大上下文窗口问题。

驱动第二层的机制通常是围绕智能体的程序性、确定性构造。正如我们将在本章后面看到的，规划可能涉及长时间运行的顺序思考服务器，状态可能存储在进程内存或数据库中，决策是代码中的检查，而工具就是智能体本身。

### 9.1.3 第3层：元循环

第三层，即元智能体循环，是控制内部第1层或第2层过程的智能体或智能体们。第2层和第3层之间的关键区别在于，第二层通常由代码指导，而元层则由一个或多个智能体控制。图9.3展示了第三元层的整合，它由智能体编排器或智能体协作控制。

![](./images/CH09_F03_Lanham4.png)

##### 图9.3 使用编排器控制器的智能体循环第3层与更自由形式的协作模式之间的比较

因为一个智能体控制第三元层，它的位置决定了循环的子类型：编排或协作。在编排控制的元循环中，单个委派智能体在其自己的内部循环中封装了_决策_、_状态_和_计划_元素，将委派子智能体用作_工具_。当决策智能体或智能体们成为循环的一部分时，它就变成了一个可能共享也可能不共享_状态_和_计划_组件的智能体协作，具体取决于用例。

因为我们已经广泛涵盖了内部智能体循环，本章剩余部分的重点将是实现更高级形式的智能体循环：第2层任务循环和两种形式的元循环（第3层）。

## 9.2 第2层：深度研究智能体的循环

深度研究智能体可能是第2层智能体循环最著名的实现，其中“深度”一词指的是智能体为完成更长周期、深入的研究目标而执行的更广泛搜索。研究智能体已经变得普遍，每个基础模型提供商（Google Gemini、Anthropic Claude和OpenAI ChatGPT）都提供某种形式的深度研究智能体。

一个好的智能体循环和一个失控的成本发生器之间的区别归结为三件事：你如何管理迭代之间的状态，你如何定义终止条件，以及你如何将智能体连接到外部工具。处理好这些，你就能得到一个可靠地收敛于高质量输出的系统。处理不好，你得到的将是一个白白消耗令牌的系统。

### 9.2.1 创建初始状态和计划

流程从初始化 `ResearchState` 和 `ResearchPlan` 对象开始，如下一个清单所示。这些对象在智能体执行外部维护外部状态和计划，并在智能体循环的每次迭代中共享。

##### 清单9.1 01_research_state_plan.py（研究状态和计划）

```python
from pydantic import BaseModel, Field

class SubTopic(BaseModel):
   """研究计划中的单个子主题。"""
   name: str
   status: str = "pending"  # pending | in_progress | complete
   notes: str = ""

class ResearchPlan(BaseModel):
   """指导研究过程的战略计划。"""
   sub_topics: list[SubTopic] = Field(default_factory=list)
   strategy_notes: str = ""

   def to_context(self) -> str:    #1
       return str(dict(
           sub_topics=[
               dict(name=st.name, status=st.status, notes=st.notes)
               for st in self.sub_topics
           ],
           strategy_notes=self.strategy_notes,
       ))

   @property
   def progress_summary(self) -> str:    #2
       if not self.sub_topics:
           return "No plan yet"
       total = len(self.sub_topics)
       complete = sum(1 for st in self.sub_topics
                      if st.status == "complete")
       in_progress = sum(1 for st in self.sub_topics
                         if st.status == "in_progress")
       return (f"{complete}/{total} complete, "
               f"{in_progress} in progress")

class ResearchState(BaseModel):    #3
   goal: str = ""
   findings: list[str] = Field(default_factory=list)
   sources_consulted: list[str] = Field(default_factory=list)
   follow_up_questions: list[str] = Field(default_factory=list)
   plan: ResearchPlan = Field(default_factory=ResearchPlan)
   iteration_count: int = 0
   max_iterations: int = 10
   status: str = "in_progress"

   @property
   def should_continue(self) -> bool:    #4
       return (
           self.status == "in_progress"
           and self.iteration_count < self.max_iterations
           and len(self.follow_up_questions) > 0
       )

   def to_context(self) -> str:    #5
       return str(dict(
           goal=self.goal,
           findings=self.findings,
           sources=self.sources_consulted,
           pending_questions=self.follow_up_questions,
           plan=self.plan.to_context(),
           iteration=self.iteration_count,
           remaining=self.max_iterations - self.iteration_count,
       ))

 #1 将计划转换为供LLM消费的字符串上下文的辅助函数  
#2 计划的一个属性，返回计划执行情况的上下文字符串摘要  
#3 一个可以从一次智能体迭代携带到另一次迭代的状态对象  
#4 确定智能体是否应继续研究的属性  
#5 将状态转换为供LLM消费的上下文字符串的辅助函数  
```

`ResearchState` 本质上是跟踪循环中每次智能体迭代的共享记忆。在第一次迭代中，状态将是空的，但随着智能体的迭代，它将快速积累信息。同样，`ResearchPlan` 持有智能体在每次迭代中将使用的策略。在此示例中，计划将随着智能体遍历观察而更新。它还增加了对如何最好地进行搜索/研究的理解。

清单9.1中的关键设计决策是 `to_context` 方法，它将累积的状态/计划序列化为智能体可以消费的字典字符串。通过控制智能体在每次迭代中看到的内容，我们可以管理上下文窗口的使用，并使智能体专注于最相关的信息。随着发现跨多次迭代累积，您可能希望总结早期的发现，而不是直接传递它们。

### 9.2.3 理解迭代主体输出

在我们跳入主要决策块之前，我们想涵盖每次智能体迭代生成的输出。迭代输出捕获循环的每个周期中发生的工作。这就是智能体感知、规划、行动和学习的地方。在代码中，迭代主体通常是单次调用 `Runner.run()`，并将当前状态/计划作为上下文，然后解析智能体的输出以更新状态。

迭代输出（`ResearchIteration`）应设计为确保智能体生成循环控制器可以解析的结构化输出。这很重要，因为循环控制器需要从每次迭代中提取特定的信息片段：智能体发现了什么，它下一步想做什么，以及它是否认为目标已达成。以下清单显示了研究迭代的强类型输出模型。此模型还包括一个也跟踪子主题更新的子对象。

##### 清单9.3 03_iteration_output.py（迭代主体）

```python
class SubTopicUpdate(BaseModel):    #1
   """对计划中单个子主题的更新。"""
   name: str
   status: str  # pending | in_progress | complete
   notes: str = ""

class ResearchIteration(BaseModel):    #1
   """单次研究迭代的输出。"""
   summary_of_findings: str    #2
   sources_used: list[str]     #2
   follow_up_questions: list[str]     #2
   goal_satisfied: bool     #2
   confidence: float     #2
   reasoning: str     #2
   plan_updates: list[SubTopicUpdate] = []       #3
   new_sub_topics: list[SubTopicUpdate] = []    #4
   strategy_notes: str = ""

 #1 使用嵌套对象是要求智能体输出结构化的极好方法。  
#2 这些字段将要求智能体通过自我评估输出关于迭代的关键指标和状态，例如目标是否以高置信度满足  
#3 将应用于下一次迭代计划的更新  
#4 智能体识别出的仍需要解决的子目标（主题）集  
```

在这里使用强类型输出是不可妥协的。没有它，你将不得不解析自由文本响应来确定循环控制流，这是脆弱且容易出错的。`follow_up_questions` 和 `goal_satisfied` 字段尤其重要，因为它们驱动下一次迭代。当这个列表为空，或目标被标记为已满足时，它向状态发出信号，表示没有剩余内容需要调查。这会触发终止。

##### 智能体的强类型输出

请记住，当我们对智能体使用强类型输出时，我们不会在智能体指令中调用输出格式。SDK负责将输出格式作为指令的一部分传递。通过让SDK管理输出类型化，我们避免了为每次类更改修改智能体指令的需要。

### 9.2.4 终止门控

终止门控是每次迭代结束时的决策点，决定循环是继续还是退出。正如我们之前建立的，终止条件应始终是分层的。以下是您可能需要考虑的一些优先级示例：

* _硬限制_ —最大迭代次数。这是您的安全网，应始终存在。设置此限制取决于您希望智能体运行多长时间以产生良好输出。
* _预算限制_ —最大令牌或成本支出。对于具有成本约束的生产系统至关重要。为了更好地控制成本，减少智能体每次迭代消耗的令牌数量。
* _目标满足_ —智能体声明其目标已达成。这是理想的退出路径，但它也是一种可能有偏见的内部自我评估。为了解决这种偏见，您可以引入另一个目标智能体来做这个决定。
* _质量阈值_ —外部评估者或智能体自身的自我评估超过最低置信度标准。由于这是一种自我评估，您可能希望使用一个质量智能体或一个结合的目标/质量智能体来控制它。
* _停滞检测_ —智能体的输出在 _N_ 次迭代中没有有意义的变化。这可以防止产生递减回报的循环。`to_context` 函数可以帮助规范化输出并跟踪主要变化。

硬限制和预算限制是防御性的，不可妥协。目标满足和质量阈值表明智能体正在做好其工作。停滞检测是针对智能体没有失败但也没有取得进展，而是无休止地改写相同发现或运行类似搜索的情况的安全阀。

#### 停滞比失败更难检测

一个失败的智能体会抛出错误或返回空结果，这两者都很容易捕获。一个停滞的智能体会返回看起来合理的输出，但与它在两次迭代前产生的输出没有实质性差异。为了检测这一点，比较连续迭代摘要之间的语义重叠。您可以通过将两组文本转换为向量嵌入，然后测量余弦相似度来检查语义重叠。如果重叠连续两次迭代超过阈值（比如85%），则终止循环并返回您已有的内容。额外的迭代没有增加价值。

#### 无约束循环的成本

一个没有硬终止条件的智能体循环是一个等待发生的生产事故。每次迭代都消耗令牌，产生API成本，并增加延迟。在开发期间，让智能体循环直到它“满意”是很诱人的，但在生产中，您必须始终设置最大迭代次数和成本上限。即使您信任智能体对收敛的判断，防御性最大值也能保护您免受模型陷入推理循环或不断搜索不存在信息的边缘情况的影响。

##### 清单9.4 04_deep_research_loop.py（智能体初始化）

```python
research_agent = Agent(
   name="Deep Research Agent",
   instructions="""    #1
You are a deep research agent.
Your goal is to thoroughly research a topic by searching for
information, reading results, and synthesizing findings.

RESEARCH PLAN:
- On your FIRST iteration (when no plan exists yet), analyze the
 research goal and create a plan by returning sub-topics in
 new_sub_topics. Each sub-topic should be a distinct aspect of
 the goal. Set status to "pending" for unstarted topics and
 "in_progress" for the one you are currently researching.
- On SUBSEQUENT iterations, update existing sub-topics via
 plan_updates (change status to "in_progress" or "complete",
 add notes summarizing what was learned). You may also add
 new sub-topics via new_sub_topics if you discover the research
 requires areas not in the original plan.
- Use strategy_notes to record high-level observations about
 your research approach or adjustments to strategy.

Each iteration, review your plan and accumulated state, identify
gaps, search for new information, and update your findings.
follow_up_questions should contain specific search queries for
the next iterations. The plan provides strategic direction while
follow_up_questions drive tactical execution.

When you believe you have sufficient information to answer
the research goal comprehensively, set goal_satisfied to true.
""",
   model="gpt-5.1",        #2
   output_type=ResearchIteration,
)

def apply_plan_updates(    #3
   plan: ResearchPlan, iteration: ResearchIteration
) -> None:
   """从迭代中将计划更改应用到可变计划。"""
   existing = {st.name: st for st in plan.sub_topics}
   for update in iteration.plan_updates:
       if update.name in existing:
           existing[update.name].status = update.status
           if update.notes:
               existing[update.name].notes = update.notes

   for new_st in iteration.new_sub_topics:
       if new_st.name not in existing:
           plan.sub_topics.append(
               SubTopic(
                   name=new_st.name,
                   status=new_st.status,
                   notes=new_st.notes,
               )
           )

   if iteration.strategy_notes:
       plan.strategy_notes = iteration.strategy_notes

#2 一个能有效执行每个子主题/任务所需工具的模型
```

清单9.4中有很多内容，所以让我们来梳理关键的设计决策。首先，`to_context` 方法只将最后五个发现传递给智能体。这是在上下文丰富性和上下文窗口管理之间的刻意权衡。在一个运行10次迭代的循环中，传递所有发现会消耗上下文窗口的很大一部分。这将减少智能体思考和工具结果的空间。通过截断到最近的发现，我们让智能体专注于最新信息，同时相信较早的发现已经被综合到较新的发现中。

接下来，循环从 `follow_up_questions` 列表的前面弹出问题，并将新问题附加到末尾。这创建了一个广度优先的探索模式。如果您想要深度优先的探索（在移动到下一个之前完全追踪每个线程），则从列表末尾弹出。广度优先和深度优先探索之间的选择是一个可调参数，应与您的研究用例匹配。实际上，我们在这里执行的是思维树搜索。

### 9.2.6 合成最终输出

在循环完成后，我们有一个发现列表，但我们还没有一个连贯的、结构良好的研究报告。这些发现是来自各个迭代的快照，可能重叠、矛盾或存在空白。为了产生一个精炼的最终输出，我们运行一个单独的合成智能体，它使用累积的状态来生成一个统一的报告。

#### 智能体循环中的关注点分离

让研究智能体在其最后一次迭代中生成最终报告是很诱人的。抵制这种诱惑。研究智能体的工作是探索和发现。合成智能体的工作是编译和结构化。当您将这些责任合并时，您会得到那些在探索期间开始优化报告结构的智能体，这会降低搜索质量。保持它们分离，每个智能体都能专注于它最擅长的领域。

### 9.2.7 何时使用智能体循环

并非每个智能体都需要循环。当任务具有以下一个或多个特征时，使用智能体循环：

* _迭代发现_ —智能体无法在单个步骤中获得所需的所有信息。研究、分析和探索任务几乎总是属于此类别。
* _渐进式细化_ —输出随每次迭代而改进。写作、摘要和内容生成可以从迭代草稿中受益。
* _批处理_ —存在一个待处理项队列，智能体一次处理一个或小批量处理。
* _条件分支_ —智能体的下一步行动取决于上一步行动的结果。诊断智能体、故障排除流程和决策树使用此模式。
* _多源聚合_ —智能体需要组合来自多个源的信息，并且相关源在执行期间被发现，而不是预先已知。

相反，对于需要单次通过的任务，避免使用智能体循环。分类、简单问答和格式转换很少从迭代中受益。为这些任务添加循环只会增加延迟和成本，而不会提高质量。

### 9.2.8 构建重复性任务循环智能体

智能体循环的第二个主要模式是重复性任务完成。与研究智能体（探索和发现）不同，任务智能体具有明确定义的工作队列并按顺序处理项目。想想批量处理发票、转换一系列文档、针对多个端点运行测试，或将记录从一个系统迁移到另一个系统。

## 9.3 第3层：多智能体编排循环

在建立了第2层任务和研究循环之后，我们现在转向第3层元循环。回顾9.1节，元循环将一个智能体而不是代码置于_决策_、_状态_和_计划_元素的控制中。在编排子类型中，单个编排器智能体将工作委派给专门的子智能体，评估其结果，并决定是再次循环还是最终确定输出。

图9.6显示了编排器控制的元循环的架构。编排器维护全局计划和状态，将单个任务作为工具调用委派给工作智能体，并评估集体输出是否满足目标。

![](./images/CH09_F06_Lanham4.png)

##### 图9.6 图显示了编排器智能体循环流程。它还显示了状态和计划如何被访问和管理，做出了哪些决策，以及任务如何被委派给工作智能体。

这与第2层循环的关键区别在于，编排器本身是一个LLM驱动的智能体。它推理接下来要追求哪个子任务，工作器的输出是否足够，以及计划是否需要调整。这使得元循环以硬编码的`for`循环无法做到的方式具有适应性。

```python
async def run_orchestrator_loop(                              #1
   goal: str, max_iterations: int = 15
) -> ResearchState:
   search_server = MCPServerStdio(
       name="Brave Search",
       params={
           "command": "npx",
           "args": ["-y", "@anthropic/brave-search-mcp"],
           "env": {"BRAVE_API_KEY": os.environ["BRAVE_API_KEY"]},
       },
   )
   async with search_server:
       workers = {                                           #2
           "Research Worker": research_worker.clone(
               mcp_servers=[search_server]
           ),
           "Analysis Worker": analysis_worker,
       }
       state = ResearchState(
           goal=goal, max_iterations=max_iterations,
           follow_up_questions=[goal],
       )
       plan = OrchestratorPlan()
       for iteration in range(max_iterations):               #3
           orch_input = dict(
               goal=state.goal,
               current_state=state.to_context(),
               plan=str(dict(
                   sub_tasks=[
                       dict(name=st.name, status=st.status,
                            notes=st.notes)
                       for st in plan.sub_tasks],
                   strategy=plan.overall_strategy,
                   focus=plan.current_focus,
               )),
               iteration=iteration + 1,
               max_iterations=max_iterations,
           )
           decision = (await Runner.run(
               orchestrator_agent, input=str(orch_input)
           )).final_output

           if decision.is_complete or \
              decision.next_action == "finalize":            #4
               state.status = "complete"
               break

           if decision.next_action == "delegate":            #5
               worker = workers.get(decision.target_worker)
               if worker:
                   worker_result = (await Runner.run(
                       worker, input=decision.task_description
                   )).final_output
                   state.findings.append(
                       worker_result.summary_of_findings
                   )
                   state.sources_consulted.extend(
                       worker_result.sources_used
                   )
                   apply_plan_updates(
                       state.plan, worker_result
                   )

           if decision.next_action == "re_plan":             #6
               for update in decision.plan_updates:
                   existing = {st.name: st
                               for st in plan.sub_tasks}
                   if update.name in existing:
                       existing[update.name].status = \
                           update.status
                   else:
                       plan.sub_tasks.append(SubTopic(
                           name=update.name,
                           status=update.status,
                           notes=update.notes,
                       ))

           state.iteration_count = iteration + 1
       return state

#2 工作智能体存储在一个字典中，以供编排器动态分派。  
#3 编排器迭代，在每一步做出关于下一步做什么的决策。  
#4 如果编排器决定目标已完成，则退出循环。  
#5 委派时，编排器将工作路由到适当的工作智能体。

```

编排器模式之所以强大，是因为它将战略推理（追求哪个子任务，是否重新规划）与战术执行（搜索和分析）分离开来。编排器从不直接接触搜索工具；它推理下一步需要发生什么，并委派实际工作。这反映了有效的人类管理者如何运作：他们计划、委派和评估，而不是自己做每项任务。

注意，编排器可以在循环中动态重新规划。如果工作器的发现揭示原始分解不完整，编排器可以添加新的子任务或重新排序现有的子任务。这种适应性正是使元循环对于复杂的、多方面的目标（尤其是那些可能在编排器处理问题时发生变化的目标）比静态的第2层循环更强大的原因。

#### 编排器开销

每个编排器的决策都是一次LLM调用。在一个15次迭代的循环中，除了工作器调用之外，还有15次额外的调用。对于预先知道分解的直接任务，带有硬编码任务分派的第2层循环更便宜、更快。保留编排器模式用于分解本身不确定或计划必须根据中间结果演变的场景。

## 9.4 构建协作式智能体循环

第3层元循环的第二个子类型是协作，其中多个智能体作为对等体参与，而不是在委派者-工作者层级中。在协作循环中，智能体共享状态并共同决定目标何时达成。没有单个编排器；相反，智能体轮流贡献、批评和完善彼此的工作。

图9.7显示了协作元循环的架构。多个智能体共享一个共同的状态对象，并循环通过贡献轮次。一个轻量级的轮次管理器控制轮到谁，但智能体本身驱动内容并决定何时停止。

![](./images/CH09_F07_Lanham4.png)

##### 图9.7 智能体协作第3层元智能体循环的组件工作流

当不同的视角改善输出时，协作模式表现出色。一个常见的配置是研究者-批评者-合成者三元组：一个智能体生成发现，另一个挑战和验证这些发现，第三个将经过验证的发现编织成一个连贯的叙述。每个智能体看到完整的共享状态，包括来自其他智能体的贡献，因此每轮都建立在集体工作之上，而不是孤立的工作。

通常，在这种类型的智能体循环协作模式中，智能体遵循一个贡献者后跟批评者（基础化、QA或另一个批评工作的智能体）的循环。高级模式可能会演化，以便智能体自身选择下一个智能体。这可能允许智能体在将其工作发送给批评智能体之前迭代其自己的工作。本质上，它允许一个第2层智能体循环存在于一个第3层协作元循环内。

```python
   confidence: float = 0.0

class CollaborationState(BaseModel):
   """协作智能体循环的共享状态。"""
   goal: str
   contributions: list[Contribution] = Field(
       default_factory=list
   )
   round_number: int = 0
   max_rounds: int = 5
   consensus_threshold: float = 0.8                          #1

   @property
   def has_consensus(self) -> bool:                           #2
       if len(self.contributions) < 2:
           return False
       recent = self.contributions[-3:]
       if not recent:
           return False
       agreeing = sum(1 for c in recent if c.agrees_goal_met)
       return agreeing / len(recent) >= self.consensus_threshold

   def recent_context(self, n: int = 6) -> str:
       recent = self.contributions[-n:]
       return str([
           dict(agent=c.agent_name, content=c.content[:500],
                critique=c.critique, agrees=c.agrees_goal_met)
           for c in recent
       ])

 #1 共识阈值——最近必须有多少比例的智能体同意目标已达成  
#2 检查最近的智能体是否已就目标满足达成共识  
```

```python
async def run_collaboration_loop(                             #1
   goal: str, max_rounds: int = 5
) -> CollaborationState:
   agents = [researcher_agent, critic_agent,
             synthesizer_agent]
   search_server = MCPServerStdio(
       name="Brave Search",
       params={
           "command": "npx",
           "args": ["-y", "@anthropic/brave-search-mcp"],
           "env": {"BRAVE_API_KEY": os.environ["BRAVE_API_KEY"]},
       },
   )
   async with search_server:
       agents[0] = researcher_agent.clone(                   #2
           mcp_servers=[search_server]
       )
       state = CollaborationState(
           goal=goal, max_rounds=max_rounds
       )
       agent_index = 0
       while (state.round_number < state.max_rounds
              and not state.has_consensus):                  #3
           current_agent = agents[agent_index % len(agents)]
           collab_input = dict(
               goal=state.goal,
               your_role=current_agent.name,
               round=state.round_number + 1,
               recent_contributions=state.recent_context(),
           )
           print(f"\nRound {state.round_number + 1}, "
                 f"Agent: {current_agent.name}")
           result = await Runner.run(
               current_agent, input=str(collab_input)
           )
           contribution = result.final_output
           contribution.agent_name = current_agent.name
           state.contributions.append(contribution)
           print(f"  Content: {contribution.content[:100]}...")
           print(f"  Agrees goal met: "
                 f"{contribution.agrees_goal_met}")

           agent_index += 1
           if agent_index % len(agents) == 0:                #4
               state.round_number += 1

       return state

 #1 运行协作循环的主要函数，使用循环轮换智能体  
#2 只有研究者需要搜索工具。批评者和合成者基于现有发现工作。  
#3 循环继续，直到达成共识或达到最大轮数。  
#4 当所有智能体都贡献过一次时，完成一轮完整的循环。  
```

对于受益于对抗性验证的任务，协作模式比单智能体循环产生更高质量的输出。批评者捕获单个研究智能体可能忽略的未经证实的主张和事实空白。合成者解决研究者与批评者通过其来回交流所揭示的矛盾。每轮都完善集体的理解。

#### 协作与编排：何时使用哪个

当目标自然分解为可以分配给专家的独立子任务时，使用编排。当子任务相互依赖并受益于多个视角时，使用协作。研究综合、文档审查和战略分析是协作的自然适用场景。数据管道处理、并行搜索和批处理转换更适合编排。如果您不确定，从编排开始（它更容易调试），并且仅在输出质量需要对抗性审查时才升级到协作。

决定在何处使用协作与编排取决于您的用例。对于简单的批处理任务，编排效果很好。对于需要更多对抗性审查的任务，请使用协作。

## 总结

* 智能体循环是驱动研究智能体、任务智能体以及任何必须迭代以实现目标而非单次回答的智能体的核心迭代引擎。
* 每个智能体循环都依赖于三个支柱：一个目标（要实现什么）、一个迭代策略（每个周期做什么）和终止条件（何时停止）。所有这三个都必须在您的设计中明确。
* 状态累加器在迭代之间携带上下文。控制智能体在每个周期看到的内容，通过随着循环进展截断、总结或压缩早期发现来管理上下文窗口的使用。
* 强类型的迭代输出（Pydantic 模型）对于循环智能体来说是不可妥协的。循环控制器需要结构化数据，如后续问题、目标满足标志和置信度分数，以管理流控制。
* 终止条件应分层：硬迭代限制（安全网）、预算限制（成本控制）、目标满足（理想退出）、质量阈值（评估者驱动）和停滞检测（递减回报）。
* MCP 服务器为循环智能体提供动态工具发现，允许智能体搜索网络、访问文件系统和调用外部服务，而无需硬编码 `@function_tool` 包装器。将 `MCPServerStdio` 与异步上下文管理器一起使用，以实现干净的生命周期管理。
* 深度研究循环遍历搜索-读取-合成周期，积累发现并生成后续问题。使用一个单独的合成智能体在循环完成后将发现编译成精炼的报告。
* 重复性任务循环处理带有瞬时错误重试逻辑的工作队列。显式跟踪已完成、失败和待处理的项目，并在重试时将先前的错误反馈给智能体。
* 当项目独立时，使用 `asyncio.gather` 的并行迭代可显著加速任务循环，但在启用之前要注意速率限制并确保任务独立性。
* 自适应迭代预算允许智能体根据任务复杂性估算和请求迭代次数，同时尊重开发者设置的硬上限。
* 永远不要在没有硬迭代限制和成本上限的情况下部署智能体循环。一个无约束的循环是一个等待发生的生产事故。
