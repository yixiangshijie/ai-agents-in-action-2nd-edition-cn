## 2.1 理解大语言模型

大语言模型（LLM）在AI领域已无处不在。由它们驱动的AI智能体日益展现出这种组合的强大力量，我们将在整本书中利用这一点。

虽然LLM常被视为黑盒，我们可能并不完全理解内部智能的工作方式，但我们已知晓足够的知识来释放其力量。在下一节中，我们将研究LLM如何驱动我们未来的智能体和智能体系统。

### 2.1.1 LLM：概率性令牌机器

从根本上说，LLM 根据学习到的下一个词元的概率来消费文本并生成文本。由于机器学习模型不理解文本，**词元化（tokenization）** 将文本转换为**词元（token）**，这些词元根据其在模型词汇表中的位置被表示为数字。模型接收这一组词元，并输出一个覆盖整个词汇表的概率分布，为每个可能的下一个词元分配一个得分。然后，一个独立的**采样（sampling）** 步骤从该分布中选择一个词元，这个词元就是输出。这种区分很重要，因为模型对给定输入产生的是相同的分布。解码策略（贪心、top-k、核采样）决定了哪个词元被输出。

![](./images/CH02_F01_Lanham4.png)

##### 图2.1 大语言模型的训练循环和推理/生成过程。左侧是模型的训练过程，在此过程中，文档被输入以进行第一阶段的LLM训练。右侧是用户输入提示词，首先进行词元化，然后输入模型。模型随后输出用于采样和生成下一个词元的概率。

如图所示，训练LLM涉及从各种来源输入文本块，模型学习预测下一个最佳或期望的词元。它通过将模型对下一个词元的预测与实际期望输出进行比较，计算损失或误差量来实现这一点。这个误差被反馈回模型以修正自身并预测期望词元。该反馈背后的机制称为**反向传播（backpropagation）**：根据计算出的损失调整模型内部权重的过程。在实践中，训练过程会在大规模文本上运行数十亿次这样的迭代。

一旦基础训练完成，模型会经历一个单独的**对齐（alignment）** 阶段，使用诸如**基于人类反馈的强化学习（RLHF）** 等方法。RLHF不会教给模型新知识或提高其推理能力。它教会模型遵循指令、提供有用的回复并避免有害输出。推理能力的提升来自不同的技术，如**思维链微调（chain-of-thought fine-tuning）** 和**过程奖励模型（process reward models）**。

这种广泛的训练产生了一个语言模型，它可以消费词元化的文本并为下一个词元生成概率。一旦模型利用这些概率进行采样并输出下一个词元，它会继续生成概率，直到遇到一个**EOS（流结束）** 词元，此时过程停止。

推理速度很快，词元快速流式输出，这给人一种智能的错觉。模型只是在一次一个地预测下一个最佳词元。使这些预测良好的是某种内部概率图。Transformer并不将知识存储为查找表。能力存在于**注意力（attention）** 和**前馈（MLP）** 层的习得权重中——数十亿个通过训练塑造的参数。每次前向传播都会将输入通过这些层，构建上下文的内部表示，并输出下一个词元的概率分布。模型每次都从头开始重新计算该预测，使用编码了所学知识的权重。

> **注意**：现代LLM在基础训练后会经历一个使用RLHF等技术的对齐阶段。RLHF不会提升推理能力。它教会模型遵循指令并以对话方式回应。推理能力的提升来自独立的技术，如思维链微调。要深入了解微调和RLHF，请参阅Sebastian Raschka所著的《Build a Large Language Model (From Scratch)》（Manning, 2024）。

这里的关键要点是理解LLM是根据预测词元的概率工作的，因此我们在构建提示词和指导智能体时，必须考虑词元和词元预测。但首先，我们需要理解什么是词元。

### 2.1.2 什么是词元？

我们使用术语**词元化**来讨论将文本分解为LLM可以消费的小块的过程。这些词元——单词和单词组成部分——随后被转换为LLM词汇表中的数值索引。这些索引数字随后被输入LLM，LLM输出词元索引的预测，经过采样后作为下一个预测词元输出。

图2.2展示了与常规文本相比，一段JSON文本的词元化过程。

![](./images/CH02_F02_Lanham4.png)

##### 图2.2 常规文本和JSON文本词元化的比较

图中显示，JSON被词元化为13个词元，而等效的文本被词元化为6个词元。这表明文本长度并不是词元数量的良好指标。

所有那些额外的大括号、引号和字段名也都被词元化，这增加了相同底层数据的词元计数。这并不意味着我们不应该在提示词中使用JSON或与LLM交换数据。它只是说明了JSON在词元数量和开销方面的代价。大多数LLM现在都能很好地处理JSON，但情况并非总是如此。

关键要点是，原始文本长度并不代表词元长度。LLM提供商分别对输入和输出词元定价，输出词元通常比输入词元贵2到5倍，具体取决于提供商和模型。这种成本在大型智能体系统中会迅速累积，单次运行可能通过LLM处理数百万个词元。

像`tiktoken`这样的Python包可以帮助你测量文本中的词元数量。你将输入文本传递给`tiktoken`，它会根据模型的词汇编码格式输出词元数量。我们稍后使用的OpenAI Agents SDK也为每次LLM交互提供了输入和输出词元的自动计数。

大量的输入词元通常也会增加LLM输出的不确定性。但我们可以通过提示词来控制这一点，并且正如我们将在下一节中看到的，还有其他方法来调整LLM的输出。

### 2.1.3 调节温度、top_p等参数

词元并不是唯一塑造模型输出的因素。我们可以向模型传递几个改变其行为的参数，其中大多数都带有权衡。例如，提高温度会使模型更具创造性，但事实准确性会降低。这些参数与身份验证和重试等API设置不同，后者控制调用本身而非其生成。图2.3总结了这些参数及其影响。

![](./images/CH02_F03_Lanham4.png)

##### 图2.3 可用于修改LLM输出的各种参数。图的右侧是模型预测和采样词元以生成输出的放大视图。左侧是我们看到的可用于更改模型内下一个词元采样的各种参数。

图的右侧显示了图2.2中LLM预测和采样块的近视图。**采样（Sampling）** 是模型从刚刚生成的概率分布中选择一个词元的步骤。采样参数（temperature、top-p、presence_penalty、frequency_penalty）都会影响该选择。max_tokens参数则不同。它限制了模型在单次回复中可以输出的词元数量。

表2.1也提供了这些参数的更详细总结。

##### 表2.1 LLM参数及其如何用于控制输出

| 参数 | 典型范围 | 实际作用 | 何时提高/降低 |
| :--- | :--- | :--- | :--- |
| **Temperature** | 0–2 (默认 ≈ 1) | 在采样前缩放每个词元的概率。较高值 ⇒ 更平坦的分布 ⇒ 更多随机性/创造性。较低值 ⇒ 更尖锐 ⇒ 更确定、重复的输出。 | 头脑风暴、创意写作时提高。一致性答案、代码生成时降低。 |
| **top-p (nucleus)** | 0–1 (默认 ≈ 1) | 仅保留累积概率 ≥ p 的最小词元集；重新归一化，然后采样。在不允许低概率“尾部”词元的情况下控制多样性。 | 降低 (< 0.9) 以获得紧凑、切题的散文。提高 (~1) 以获得自由形式的生成。 |
| **max_tokens** | 1–8192 + (依赖模型) | 模型在回复中可生成词元数量的硬性上限。(提示词词元 + max ≤ 上下文窗口) | 设置为略高于预期长度。防止冗长并降低成本。 |
| **presence_penalty** | –2 … +2 | 一旦词元出现则增加惩罚；鼓励新话题 | 使用 0.6–1.0 避免重复已提及的概念。 |
| **frequency_penalty** | –2 … +2 | 按词元的当前计数比例进行惩罚；阻止短语重复 | 使用 0.2–0.8 减少长回复中的重复或循环。 |

我们经常根据智能体的角色来调整这些参数。一个编码智能体，其输出需要严格一致，可能会将温度设置为0并使用低频惩罚，以便常见语法（括号、关键字、变量名）可以自由重复。一个创意写作智能体可能会将温度设置为2并应用高存在惩罚，推动模型引入新想法而非循环相同的内容。

同样重要的是要理解这些参数是影响模型，而非施加严格限制。将温度设置为0并不能保证LLM总是输出相同的答案。有些参数还会相互作用，使得同时调整两者变得不可预测，这就是为什么像OpenAI这样的提供商建议只调整温度或top-p，而不是两者都调。

在大多数情况下，你只会修改温度（基于智能体的角色）和max_tokens。设置max_tokens限制回复长度，避免昂贵的交互，并作为防止意外长输出的护栏。种子参数也值得了解。当你传入相同的种子、提示词和参数时，模型返回相同的输出，这对于调试和评估运行很有用。其他参数通常可以保留默认值，我们依靠提示工程来控制模型，如下一节所示。

## 2.2 通过提示工程控制LLM（智能体人格）

有些人将提示工程视为与LLM相关的炒作。事实恰恰相反。提示工程是让你使智能体可预测、高效、避免意外行为并控制成本的方法。随着模型的改进，该领域的范围已经发生了变化。现代LLM可以处理比需要仔细提示的旧模型多得多的内容。但核心技能保持不变：编写清晰、结构良好的提示词，使智能体可以可靠地执行。如第1章所述，人格或提示词是智能体的核心功能层，而提示工程就是确保该层正确的实践工作。

### 2.2.1 应用核心提示词技术

你如何向LLM提示——语言、关键术语和语调——可能因LLM而异，但有几种关键技术已被证明是通用的。这些技术在各种LLM上都能很好地工作，因为它们利用了所有模型在某种程度上共享的基础训练模式。通过遵循这些模式，我们还可以构建通用的提示词和智能体，它们可以在不同的LLM之间迁移而无需修改。

表2.2总结了这些技术，并附有示例和它们如何应用于驱动智能体的智能体人格或基本指令集的描述。它介绍了提示工程的一般技术。我们可能会发现某些方法，如“定义确切的输出格式”，在本书中不适用，因为我们使用OpenAI Agents SDK，它会自动处理这一点。

##### 表2.2 基本的提示词技术

| :--- | :--- | :--- | :--- |
| **分配明确的角色或人格** | 设定一个能确定词汇、语气和深度的角色。 | “你是一位高级DevOps工程师。诊断下面的Docker错误……” | 切换人格字符串以创建研究员、评论家或规划智能体，无需更改代码。 |

### 2.2.2 像LLM一样思考

一个常用于描述编写有效提示词的类比是，编写提示词就像你是在给一个新的_某某_下达指令。这里的_某某_可以是新员工、研究员、作家，或任何缺乏执行任务背景经验的人。这并不意味着你必须编写一本手册。尽管如此，你希望避免假设，并在描述任务时尽可能具体。例如，写一个关于时间旅行的故事，将旅行者带回1921年，不超过五段，使用第一人称、幽默且引人入胜的语调。

这也意味着你可以按照你希望智能体执行的方式描述任务。例如，在执行任务_X_时，始终记得考虑_Y_和_Z_。通常，LLM会按此执行。更进一步，你可以在提示词中描述整个工作流，包括决策，智能体将能够执行它们。

### 2.2.3 避免常见的提示词陷阱

表2.3强调了你为智能体编写提示词指令时，在过度设计提示词时可能遇到的几个常见陷阱。

##### 表2.3 过度设计提示词时遇到的常见陷阱

| 问题 | 描述 | 解决方案 |
| :--- | :--- | :--- |
| **过于复杂** | 一个冗长、多主题的提示词可能压倒LLM的上下文窗口，增加词元成本，并产生冲突的指令。 | 将提示词分解为更小的、特定于角色的提示词（或智能体步骤）。每个步骤在移交给下一个角色之前处理一个明确的目标。 |
| **矛盾** | 提示词包含冲突的指令（例如，“简洁”和“深入解释”）。模型必须猜测遵循哪条规则。 | 从头到尾通读提示词，就像你是模型一样；移除或重新表述冲突。使用项目符号列表使优先级明确。 |
| **过于简单** | 一个单一的微提示词触发多次往返和网络调用，增加延迟和成本。 | 将相关的微任务合并到一个更丰富的提示词中，或授予智能体有限的决策权（例如，工具使用、函数调用），以便它能在内部处理分支。 |
| **分隔符不一致** | 在一个提示词中混合使用“”” … “””、<<< … >>> 和 markdown 围栏会混淆模型和下游解析器。 | 为每个提示词采用一种分隔符样式（例如，<<<BLOCK … BLOCK>>>），并在你的提示词指南中记录它。 |
| **过于明确** | 在一个提示词中塞入几十条规则或约束会增加长度，并增加某些规则被忽略的风险。 | 只保留关键约束。如果必须强制执行许多规则，将它们分散到链式提示词中，或添加一个验证和纠正输出的护栏步骤。 |
| **输出可变** | 你已将温度设置为0，但在给定相同输入时，你的提示词回复仍然各不相同。 | 将温度设置为零只会减少变异性。如果你需要提示词始终保持一致，请确保指令明确，使用分隔符，并且不使用相反的指令。 |

当然，在你构建强大智能体的旅程中，你会遇到其他必须解决的陷阱。使用LLM帮助你编写和重写提示词可能非常有用，但要注意提示词不要变得过于复杂或明确。最重要的是，如果你的杰作提示词变得庞大或过度复杂，不要害怕将其分解为更小的任务。

现在我们对LLM及其提示方式有了清晰的理解，我们将在下一节中将这些原则应用于构建智能体。

## 2.3 使用OpenAI Agents构建智能体

OpenAI Agents SDK是在几个早期的智能体框架之后出现的，并受益于这些经验教训。结果是这个SDK轻量、可扩展且灵活。我们将使用这种模式在本书中构建所有智能体，从下一节中的第一个智能体开始。

### 2.3.2 设置智能体模型和其他参数

清单2.5显示了设置智能体模型和模型参数的示例。我们希望此智能体的规划保持一致，因此我们将温度设置为0。在温度0时，模型在每一步选择概率最高的词元，而不是采样，这会在多次运行中产生相同的计划结构，并使智能体的行为更容易调试和评估。我们将其他参数保留为默认值，并指定了我们想要使用的确切模型；否则，SDK将默认使用提供商当前指向的任何模型。

> **关于推理模型的说明**
> OpenAI、Anthropic等公司的最新模型暴露了一个`reasoning-effort`参数（有时称为`thinking`或`reasoning_effort`），用于控制模型在回答前进行多少内部思考。默认值通常是中等或高。对于智能体来说，这通常是错误的设置。智能体通常运行许多小的、范围明确步骤，其中提示词和工具已经约束了工作。高推理努力会增加延迟和词元成本，而不会改变答案。对于常规智能体步骤，将推理设置为无或最小，并将更高的努力保留给需要它的特定步骤（规划、复杂工具选择、调试卡住的循环）。这使智能体在默认情况下快速且廉价，而将深思熟虑保留给值得的时刻。

### 2.3.3 控制输入和类型化输出

图2.5显示了一个简单的智能体工作流，并比较了没有输入/输出类型的工作流与使用输入/输出类型的智能体工作流。当智能体使用类型化输出时，这些输出可以输入到另一个智能体，而无需担心回复的变异性破坏工作流。记住，LLM是概率性的词元预测器，输出总是有可能变化。通过对输出和输入进行强类型化，我们避免了破坏或混淆智能体工作流。

![](./images/CH02_F05_Lanham4.png)

##### 图2.5 比较智能体工作流在有和没有类型化输出/输入的情况下如何运行。在上面，智能体不使用严格输出，可能以任何方式回复，这增加了将输出传递给下一个智能体时的变异性。相反，在下面，智能体提供严格输出，这减少了变异性，并提高了接收输出的智能体或流程的清晰度。

清单2.7显示了我们从前面的示例中为支持类型化输出而必须应用于规划智能体的更改。这里，我们使用Pydantic类型库来定义基础数据模型。我们现在期望智能体输出一个任务列表。

我们希望为每个任务提供更多信息，因此我们不使用列表，而使用字典，如下一个清单所示。

##### 清单2.8 04_output_types.py

```python

   tasks: dict[int, str]    #1
   """A list of tasks to perform for research."""

```

运行此代码，你将看到以下错误：

> _Exception has occurred: UserError_ 

> _Strict JSON schema is enabled, but the output type is not valid. Either make the output type strict, or pass output_schema_strict=False to your Agent()_ 

该错误发生是因为我们的类型没有严格定义，这意味着底层传递的JSON结构可以被修改。理解这些错误的含义以及如何解决它们至关重要。你可以通过两种方式解决这些错误：

* 将类型包装在允许你禁用严格JSON检查的输出模式中，如下所示（不推荐）：

```python
AgentOutputSchema(ResearchPlanModel, strict_json_schema=False)
```

* 找到并解决问题。ChatGPT或Claude可以很好地帮助你找到并解决问题（推荐）。

清单2.9显示了解决方案，即找到并替换有问题的类型为允许JSON类型保持严格的代码。问题是`dict[int, str]`对象允许额外属性，违反了严格JSON模式的标准。通过将代码输入ChatGPT并让其搜索和推荐解决方案，问题得到了解决。避免禁用严格JSON（`strict_json_schema=False`），因为这可能为难以发现的错误和其他问题敞开大门。

##### 清单2.9 05_output_types_fixed.py

```python
from pydantic import BaseModel, ConfigDict
from typing_extensions import TypedDict

class Task(TypedDict):    #1
   id: int
   description: str

class ResearchPlanModel(BaseModel):
   tasks: list[Task]    #2
   """Numbered tasks for research."""

   model_config = ConfigDict(extra='forbid')    #3

OUTPUT
tasks=[{'id': 1, 'description': '回顾学术文献'}, {'id': 2, 'description': '分析近期案例研究'}, {'id': 3, 'description': '采访行业专家'}, {'id': 4, 'description': '参加AI会议'}, {'id': 5, 'description': '探索在线课程'}]

 #1 解决方案是用TypedDict对象替换dict[int, str]。  
#2 任务现在在Task对象列表中  
#3 我们还设置模型对象禁止添加额外属性，保持严格。  
```

类型化输出的另一个关键好处是，你不需要在提示词中定义输出格式。虽然你经常会在提示词中描述你想要的总体输出，但你不需要严格解释格式。这简化了你的提示词，使它们更通用，更不容易出错。

## 2.4 通过工具集成增强智能体

当智能体拥有自主性，即做出和执行决策的能力时，它们的工作效果最好。如果我们将此规则应用于我们当前的智能体——研究规划器，我们可以看到它还不是一个智能体。到目前为止，我们开发的只是一个提示词步骤。构建带有提示词步骤的AI工作流是可行的，这被称为**提示词链式调用**。

我们希望开发灵活且适应性强的智能体，这需要自主性。赋予智能体自主性的最简单方法是提供它们可以决定何时何地使用的工具。在下一节中，我们将研究为智能体添加工具。

### 2.4.1 为智能体提供工具

工具是智能体可用的能力。行动是智能体用这些能力做的事情。工具可以是智能体可以调用的代码函数、通过MCP等协议暴露的外部服务，或交给另一个智能体。我们可以在内部构建工具或消费外部工具。第3章将探讨智能体可以通过MCP消费的许多工具。

图2.8显示了智能体可用于消费工具的模式。在本章中，我们专注于同一代码库内的内部使用。在第3章中，我们将更深入地了解MCP以及我们如何使用它来消费外部工具。MCP允许工具和工具服务器在本地运行或远程托管并通过HTTP访问。

![](./images/CH02_F08_Lanham4.png)

##### 图2.8 智能体可用于消费工具的各种模式。工具可以是内部代码函数，也可以是通过MCP协议连接的、本地或远程托管的外部连接。

该图展示了智能体如何执行工具、工具调用是什么样的，以及这些工具可能位于何处。工具可以是本地的（在代码库内）、通过进程托管或通过HTTP外部托管。这种灵活性为我们提供了许多构建和选择智能体将消费的工具的选项。

清单2.11显示了我们如何用`get_research_sources`工具扩展我们简单的研究规划智能体。我们可以添加该工具，但更好的做法是在提示词指令中明确指出何时使用它。更新后的提示词现在指示智能体首先提取可用的研究来源，并制定一个将这些来源纳入每个特定任务的计划。

对于此示例，我们返回可用研究来源的静态列表，但在更实际的场景中，我们会查询可用资源。然后智能体可以制定计划并仅搜索这些来源。使用这样的模式使我们的智能体从基础上具有可扩展性。稍后，我们可以在工具内开发查询机制以提供动态来源列表。

我们可以使用`@function_tool`装饰器轻松暴露任何内部函数。此装饰器创建所有内部管道，并生成智能体将用于决定如何执行工具的JSON工具描述。重要的是要记住，在你向智能体注册工具后，它会被包含在对LLM的所有调用中，无论该工具是否被使用。

虽然你可以为单个智能体提供数十个工具，但工具数量会影响智能体选择正确工具的能力。现代前沿模型比早期模型能处理更多的工具，实际上限取决于工具之间的区别程度、它们描述的清晰程度以及模型的能力。与其固定一个数字，不如围绕驱动限制的约束进行规划。以下几个值得了解：

* _工具是开销_。你向智能体注册的每个工具的JSON描述都会在每次LLM调用时发送，正如我们在2.1.2节中看到的JSON词元化，这种结构并非免费。每个工具描述都会增加输入词元，无论工具是否被使用，这些词元都在每次调用时付费。对于每个任务进行多次调用的智能体来说，成本会累积。
* _工具增加复杂性_。你向智能体注册的每个工具都会在提示词中添加描述如何使用该工具的指令。更多的指令和更长的提示词会增加变异性以及系统出现故障的可能性。
* _工具可能失败_。每个工具都增加了另一个潜在故障点和另一个智能体可能需要恢复的表面区域。健壮的智能体工作流最小化故障点，并为确实发生的故障做好规划。常见模式包括：对瞬时故障（网络超时、速率限制）进行退避重试、智能体可以读取和响应的结构化错误返回（而不是崩溃循环的原始异常）、当主要工具不可用时的后备工具或替代路径，以及超时设置以防止挂起的工具无限期地阻塞智能体。
* _工具就是决策_。当智能体使用工具时，它做出了一个决策，这既可以是强大的，也可以是有问题的。允许智能体决定搜索某些内容是可以的，但给它一个可以删除文件的工具可能不明智。要意识到你授权智能体做出的决策，并对其做出错误决策的结果感到放心。如果你允许智能体删除文件，就要预期它会删除它有权访问的所有文件。

工具是智能体的强大补充，但它们也提供了许多误用机会和故障点。因此，将智能体的工具使用限制在它所需要的范围内至关重要。幸运的是，OpenAI通过追踪提供了一种有用的方法来识别工具使用中的问题。

### 2.4.2 追踪智能体工具使用

智能体工具使用可能迅速变得复杂，而OpenAI Traces页面对于调试问题、提高性能和优化工作流来说可能是无价的。清单2.12显示了一个更新后的代码示例，我们的Planning Agent现在使用多个工具。该智能体现在有两个函数，其中一个依赖于调用前一个函数。

##### 清单2.12 08_agent_with_tools_tracing.py

```python
instructions = """
你是一位研究规划助手。

**任务指令**
- 你将获得一个研究主题。
- 首先使用工具 get_research_sources() 
获取可用研究来源的列表。 
- 使用工具 get_resource_url()    #1
查找研究来源的URL。
- 将你的研究计划限制为仅使用可用的研究来源。
- 你的任务是提供研究该主题的计划。
- 输出5个简洁的任务，并指定每个任务将使用哪些可用的研究来源。
"""

class ResearchSource(TypedDict):    #2
   name: str
   """Name of the search source."""
   url: str
   """URL of the search source."""


class Task(TypedDict):
   step: int
   """Task Step."""
   research_source: ResearchSource    #2
   """The source to search."""
   description: str
   """Task description."""

class ResearchPlanModel(BaseModel):
   tasks: list[Task]
   """Numbered tasks for research."""
   model_config = ConfigDict(extra='forbid')


@function_tool
def get_research_sources() -> list[str]:
   """Provides a list of research sources."""
   search_sources = [
       "Wikipedia",
       "Google",
       "YouTube",
   ]
   return search_sources   

@function_tool
def get_resource_url(research_soruce: str) -> str:    #3
   """Provides a url for the research source."""
   search_sources = {
       "Wikipedia": "https://www.wikipedia.org",
       "Google": "https://www.google.com",
       "YouTube": "https://www.youtube.com",
   }
   return search_sources[research_soruce] 

agent = Agent(
   name="Research Planner", 
   instructions=instructions,
   output_type=ResearchPlanModel,
   tools=[get_research_sources, get_resource_url],    #4
   )

input = "了解AI智能体"

with trace("深度研究工作流 (工具)"):
   result = Runner.run_sync(
       agent, 
       input=input,
       )

print(result.final_output)

 #1 指示智能体使用 get_resource_url 函数获取资源的URL。  
#2 我们添加一个子类型来保存研究来源的详细信息。  
#3 返回研究来源URL查找的新函数  
#4 将新函数添加到智能体可以使用的工具列表中  
```

现在指示智能体首先获取研究来源列表，然后使用新的`get_resource_url`工具查找每个来源的URL。这意味着智能体必须进行多次工具调用，其中一些依赖于初始工具调用。虽然这个示例是人为设计的，但它演示了智能体工具使用中的典型模式，称为**工具链式调用**。

当智能体使用一个工具的输出驱动另一个工具的输入时，就会发生工具链式调用。在某些情况下，提示词指令定义了该链的外观。更常见的是，我们让智能体决定哪些工具链式调用到哪些。

现代前沿模型还可以在单次回复轮次中并行发出多个工具调用，当这些调用相互独立时。一个研究主题的智能体可能会同时调用搜索工具、日历工具和知识库查找，然后对组合结果进行推理。与顺序链相比，并行调用显著减少了延迟，但仅当工具不依赖于彼此的输出时才有效。智能体（或设计提示词的开发者）决定哪种情况适用。

OpenAI Traces页面提供了一种极好的方式来查看这种复杂的工具链。图2.9是Traces页面的截图，显示了清单2.12中智能体的示例追踪。

![](./images/CH02_F09_Lanham4.png)

##### 图2.9 OpenAI Traces页面显示了智能体执行的各种工具调用和LLM回复

如图所示，所有智能体调用工具或提示LLM的调用都被跟踪。这使我们能够通过查看事件顺序并检查工具调用的输入和输出来追踪工具链。从Traces页面，我们还可以高层地看到何时进行每个工具或提示词调用以及花费了多少时间。

这种优秀的追踪在使用OpenAI API时是包含在内的。如果你想使用不同的API，你仍然可以使用Traces页面，但你需要注册API并使用密钥。这是本书中不会涉及的高级场景。

正如我们在本章中看到的，智能体可以很容易地设置和快速运行。但它们也可能很快变得复杂。为了帮助管理这种复杂性，像Traces页面这样的工具对于理解我们的智能体在做什么以及何时做是非常宝贵的。

## 总结

* 大语言模型是概率性的词元预测器。理解词元化和概率驱动有效的成本、上下文和质量控制。
* 文本长度不等于词元长度：测量词元（例如，使用tiktoken或Agents SDK遥测）以保持预算和上下文窗口在可控范围内。
* 生成旋钮，如温度、top-p、max_tokens和惩罚项，允许你为每个智能体角色权衡创造力、一致性和成本。
* 精心制作的提示词遵循明确的人格、前置指令、结构化分隔符、少样本示例和思维链的基本规则，引导LLM产生可靠的、符合规格的输出。
* 结构良好的提示词避免了常见的陷阱（过度复杂、矛盾、分隔符模糊或输出可变），并使智能体更安全、更便宜。
* OpenAI Agents SDK将一个提示词变成一个可运行的智能体；你可以固定特定的模型和参数设置以匹配智能体的任务特征。
* 类型化的输入/输出模式（Pydantic）消除了脆弱的字符串解析，并在LLM的随机性本质下保持多智能体工作流的稳定。
* OpenAI API中的内置追踪暴露了每次LLM交互和工具调用，为调试和优化提供了有价值的可见性。
* 授予智能体工具——现在是本地函数，稍后是MCP托管的服务——提供了真正的自主性；限制工具列表以减少词元开销和故障风险。
* 工具链式调用允许智能体自主地排序多个工具调用；追踪数据揭示了决策路径并突出了性能瓶颈。
* 结合提示工程、模型调整、类型化模式、追踪和策划的工具集，产生了能够自信地规划、推理和执行深度研究任务的生产就绪智能体。
