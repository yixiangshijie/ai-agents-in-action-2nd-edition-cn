### 11.1.1 核心层：人格

你的智能体的人格塑造了所有下游选择（工具、记忆、规划）。将人格视为一个 API 契约，而不是一段散文。一个精心设计的人格（带有清晰的提示指令）可以在各个层面显著影响你的智能体的行为。在构建智能体核心（即人格）时，请记住以下四条准则：

* _定义清晰的角色和边界_。说明智能体_能做什么_和_不能做什么_；在指令的顶部和底部重复关键规则，以对抗近因效应和漂移。提供一个明确的“我不知道”的退出路径。
* _保持范围狭窄_。专门的、单一职责的智能体比“万能”机器人表现更好，并且更容易测试和成本优化。
* _优先使用结构化输出_。当智能体与其他代码或智能体交互时，返回符合模式（Pydantic/数据类）的JSON响应。这可以减少解析错误和令牌膨胀。
* _使用动态指令_。通过函数注入运行时事实（日期、组织、用户），而不是硬编码。注意不要过度使用动态注入；确保这些事实与智能体的用例和目标相关。过度注入可能导致令牌膨胀、智能体混淆和响应不佳。

以下清单整合了这四个原则，并提供了如何一致地使用它们来构建智能体的示例。

##### 清单11.1 01_core_persona_tips.py

```python
from pydantic import BaseModel, Field
from agents import Agent, Runner
from datetime import date

class Answer(BaseModel):    #1
   """Machine-readable answer format."""
   answer: str = Field(..., description="Concise, user-facing answer")
   citations: list[str] = Field(default_factory=list)
   confidence: float = Field(..., ge=0, le=1)

def core_instructions(ctx, agent) -> str:    #2
   today = date.today().isoformat()    #3
   return (
       f"You are SupportMentor, a helpful domain assistant. Today is {today}.\n"    #4
       "Always:\n"
       "1) Answer concisely; 2) Prefer retrieved context; 3) If context is missing, say you don't know.\n"
       "Output must be valid JSON matching the Answer schema.\n"
       "Never: make policy exceptions or invent facts."
   )

core_agent = Agent(
   name="SupportMentor",
   instructions=core_instructions,   
   output_type=Answer                
)

input = "What's our refund window for accessories?"
result = Runner.run_sync(core_agent, input)
print(result.final_output)

 #1 使用类型化数据类定义智能体的输出
#2 保持智能体范围狭窄且聚焦
#3 在提示指令中使用动态输入
#4 保持人格角色聚焦且定义良好
```

这个简单的代码示例展示了这四个原则的实际应用。在创建你的智能体时使用这四条准则。

### 11.1.2 工具与智能体行动

工具是智能体的手和感官。好的工具可以赋能智能体，而糟糕的工具同样会使智能体效率低下。在构建工具、从其他MCP服务器消费工具或将智能体用作工具时，有五条基本准则需要遵循：

* _使用单一职责的工具，并配以清晰的文档字符串和类型化参数_。模型从描述和模式中学习使用工具。工具的描述、名称和输入必须提供关于如何使用它的清晰指导。
* _优先为你的代码和API使用函数工具_。装饰器从类型提示/文档字符串自动生成模式，使其更易于测试和复用。
* _控制工具何时运行_。使用模型/工具设置为某些轮次强制或禁止工具；当单个API结果足够时，在第一个工具处停止。在智能体的提示中概述何时以及如何使用工具以支持此模式。
* _在合理的情况下使用预构建工具（代码、网络、文件搜索）_。使用托管工具和加固的MCP服务器来加速能力开发。请务必尽职调查，确保你消费的任何MCP服务器是安全、健壮且聚焦的。
* _为故障做计划_。添加超时/重试和友好的回退。工具可能失败，因此确保智能体理解工具失败的原因以及潜在的正确方法。

以下清单显示了一个简单的工具使用智能体，它整合了所有五条准则并展示了最佳实践。

##### 清单11.2 02_tool_action_tips.py

```python
from agents import (
   Agent, 
   Runner, 
   function_tool, 
   ModelSettings
)
import json

def _failure_error_function(context, e) -> str:        #1
   return json.dumps({"status": "error", "message": str(e)})

@function_tool(
   failure_error_function=_failure_error_function)    #1
def lookup_order(order_id: str) -> dict:   #2
   """Check order status by ID. Use when the user asks about their order."""         #2
   if not order_id.startswith("ORD-"):
       raise ValueError("Invalid order id format")
   return {"status": "shipped", "eta_days": 3}


tooling_agent = Agent(
   name="ToolingAgent",
   instructions="Use tools when needed; respond with JSON.",    #3
   tools=[lookup_order],
   model_settings=ModelSettings(
       tool_choice="auto",    #3
       parallel_tool_calls=True,     #3
   )
)
print(Runner.run_sync(
   tooling_agent, 
   "What's the status of order ORD-123?"
).final_output)

 #1 通过捕获格式错误并从工具错误中优雅恢复来为故障做计划
#2 具有清晰、定义良好的文档字符串和类型参数的单一职责工具
#3 控制工具何时以及如何运行
```

构建和消费工具时的关键要点是确保它们聚焦且定义良好。这同样适用于MCP服务器。记住，智能体消费的每个工具都会增加智能体进行的每次LLM调用的开销。

##### 工具膨胀

作为规则，始终保持工具集小巧，在使用MCP服务器时，注意服务器暴露了多少工具。记住，每个工具都附带如何使用它的描述，这会向LLM消息添加令牌和上下文。为了避免混淆你的智能体并降低成本，保持工具聚焦于智能体的角色。

### 11.1.3 推理与规划

多步骤工作需要计划、循环和停止条件。推理和规划是智能体行为的基石，可以决定目标的实现与失败。尽管如今大多数基础消费模型使用推理，但通常最好理解并控制如何使用这种推理。以下技巧可以帮助你在智能体中使用推理：

* _对非平凡任务采用ReAct_。在思考和行动之间交替；检查观察结果；然后继续。使用ReAct或思维链提示指令模式来引导智能体的思考过程。
* _对大型目标先计划后执行_。起草一个检查清单，并使用顺序思考服务器等工具记录计划。允许你的智能体构建思考给它时间思考并更好地解决复杂任务。
* _限制迭代次数_。限制轮数以防止失控循环；以有用的消息安全失败。
* _在最终输出前进行自我审查_。添加一个快速的反思性传递以捕获明显错误。

对于简单的一次性任务，你可能希望完全避免在智能体的指令中或在LLM内部进行推理，方法是关闭或降低模型的推理能力。

让我们在一个简单的规划智能体中看到这些技巧的实际应用，该智能体可以访问参考MCP顺序思考服务器。

##### 清单11.3 03_reasoning_planning_tips.py

```python
from typing import Literal
from pydantic import BaseModel
from agents import Agent, Runner
from agents.mcp import MCPServerStdio

MAX_TURNS = 3  #1

thinking_srv = MCPServerStdio(    #2
       name="sequential-thinking",
       params={
           "command": "npx",
           "args": ["-y", "@modelcontextprotocol/server-sequential-thinking"],
       },
   )

class Answer(BaseModel):
   status: Literal["ok", "needs_followup"] = "ok"
   checklist: list[str] = []
   summary: str
   sources: list[str]

instructions = f"""
You are a planner-executor.
If the message starts with 'PLAN ONLY', 
return only the checklist (no actions).
1) Plan→Execute: draft a concise 3–5 item checklist, then execute.    #2
2) ReAct: Thought→Action(tool)→Observation;    #3
inspect each observation before continuing.
3) Limit: stop after at most {MAX_TURNS} tool calls;    #1
if reached, set status='needs_followup' and return best effort.
4) Self-review: before final, catch obvious mistakes,     #4
ensure the question is answered, and cite sources.
Use seq_think between actions to decide the next step. 
Return Answer JSON only.
"""

async def main():
   agent = Agent(
       name="Planner", 
       instructions=instructions, 
       mcp_servers=[thinking_srv], 
       max_turns=MAX_TURNS,
       output_type=Answer)

   async with thinking_srv:
       question = "Summarize our refund policy and cite relevant internal docs."
       plan = await Runner.run(agent, f"PLAN ONLY. Question: {question}")
       print(plan.final_output.checklist)

       result = await Runner.run(agent, question)
       print(result.final_output)

if __name__ == "__main__":
   import asyncio
   asyncio.run(main())

 #1 限制智能体可以执行的迭代次数
#2 计划然后执行大型目标
#3 采用ReAct和CoT或其他推理策略
#4 在最终输出前进行自我审查
```

鼓励和控制推理与规划的能力可以限制LLM的外部思维。如今，大多数商业LLM具有内置推理能力，这对于一般问题效果很好。但如果不受控制，推理模型可能会变得相当不可预测。不要假设具有推理能力的LLM总是会做正确的事情；很多时候它不会。

### 11.1.4 知识与记忆

检索对于知识和记忆至关重要，两者都依赖于快速定位相关信息的能力。当我们考虑优化知识或记忆时，我们总是从检索开始。在处理检索和查找相关信息的实践时，请遵循以下一些关键约定：

* _将RAG视为一等公民能力_。在你的知识库（向量存储加元数据过滤器）上添加检索工具，而不是将文档塞入提示词中。
* _使用会话与长期记忆_。将会话线程用于短期上下文；将用户事实和偏好持久化在向量/数据库存储中，并有选择地检索它们。积极修剪。
* _在规模上优化检索_。使用ANN索引（HNSW/IVF），按领域分片，并按元数据（例如，产品、版本、日期）过滤。
* _采用混合搜索模式_。不要依赖单一模式，如向量（语义）搜索。相反，用关键词搜索、图层次搜索和其他模式来增强你的策略。
* _有意识地选择嵌入_。平衡质量、维度和成本；考虑领域特定或多语言变体以优化你的方法。嵌入大小在分配存储时可能至关重要。
* _明智地分块并基础化答案_。偏好语义连贯的块，并指示回答者_仅_使用检索到的上下文（当答案不存在时说“不知道”）。
* _保持数据新鲜并按角色和类型分区_。定期更新你的知识库，并隔离敏感部门、内容或租户。不允许用户访问他们不应访问的上下文。即使你认为你基本的单索引RAG将支持你的用例，也要预期它会改变，并计划实现角色/类型分区。

清单11.4在一个使用知识和记忆的简单单智能体示例中展示了这些最佳实践。代码假设了一个通用的向量数据库，因此你的实现可能会有所不同。

##### 清单11.4 04_knowledge_memory_tips.py

```python
from agents import Agent, Runner, SQLiteSession, function_tool

kb = load_vector_kb(     #1
   embedding_model="text-embedding-3-small",       #2
   index="HNSW",                                
   shards=["product", "policy", "engineering"], )

@function_tool
def retrieve(query: str, product: str|None=None, version: str|None=None,
            date: str|None=None, role: str|None=None, tenant: str|None=None) -> str:
   """Return grounded snippets using metadata filters."""
   return kb.query(query=query, k=5,
       filters={"product": product, "version": version, "date": date,
                "role": role, "tenant": tenant})    #3

@function_tool
def remember(user_id: str, fact: str) -> str:
   """Persist long-term user fact in vector memory; prune aggressively."""
   kb.upsert(text=fact, metadata={"user_id": user_id, "kind": "preference"}, 
   ttl_days=90); return "ok"    #3

agent = Agent(
   name="Support",
   instructions=("Use ONLY retrieved context; cite chunk ids; if absent #4
   say 'I don't know'."),    #4
   tools=[retrieve, remember],
)
session = SQLiteSession("u42-chat")    #5

kb.ingest("docs/*.md", chunking="semantic",
         partition_by=["role", "tenant", "product", "version"])    #3
print(Runner.run_sync(
   agent, 
   "Can I use the beta API on v3.2? (tenant=acme)", 
   session=session))

 #1 将RAG作为智能体的一等公民特性
#2 选择并优化嵌入
#3 保持数据新鲜并按角色分区
#4 基础化答案并确保智能体使用检索到的知识
#5 使用会话和长期记忆以获得更好的结果
```

记忆不仅可以作为记住用户对话的基础。它还可以包括来自其他智能体甚至对话的记忆和经验。知识是改善智能体上下文的关键，但一个完整的记忆解决方案可以帮助智能体随着时间的推移学习和适应。

### 11.1.5 评估与反馈

你不改进你不测量的东西。在你所做的一切的核心，记录和测量你智能体内部和外部的所有内容。这可以帮助调试和事件解决，也可以帮助你优化和改进你的智能体。你可能一开始不想构建这里所有的建议，但你会想要一个最小设置，包括基本日志记录、测试和简单的护栏。更完整的指南是生产系统的最佳实践。

使用以下最佳实践来帮助你在智能体中使用评估和反馈：

* _追踪一切_。记录提示词、工具调用、令牌计数、延迟和结果。使用SDK的内置追踪仪表板或Phoenix以获得全局视图。将日志连接到你的可观测性栈。
* _自动化评估_。维护策划的测试集和LLM作为评判者的检查；在每次提示词或模型更改时运行它们，以跟踪准确性、安全性和解决率。同样，Phoenix可以处理所有这些评估。
* _人在回路（HITL）_。人类是你最好的反馈来源。添加捕获人类用户反馈的系统，然后详细评估并优化。
* _护栏和审核_。在输入和输出以及高风险工具周围强制执行内容政策和模式验证。批评者和护栏智能体可以帮助识别不良输出并相应地处理。
* _像DevOps（AIOps）一样迭代_。测量和捕获一切，观察它，然后优化。记得包括工具调用和你的智能体可能依赖的系统（如数据库、索引和其他相关组件）的其他系统交互。

## 11.2 构建客户支持智能体的技巧

我们现在将研究当今最流行的智能体模式的特定技巧，从客户支持智能体开始。这种形式的智能体通常用于协助客户解决支持问题，但它也可以轻松适应支持内部角色。如果做得好，这些智能体可以通过分类支持工单、回答基本问题和升级关键事件来增强昂贵的客户支持角色。

对于这种类型的智能体，五个智能体层分别映射到：人格 = 支持角色；工具/行动 = API/查找/升级；推理/规划 = 分类逻辑和HITL升级；知识/记忆 = 手册的RAG；评估/反馈 = 日志/HITL。

支持智能体需要基础化答案、安全行动和优雅升级。这些智能体通常处于客户支持、反馈和投诉的前线。因此，你应该通过以下准则使它们健壮、专业且有用：

* _缩小职责范围_。从一组有限的意图（订单、退货、状态）开始，然后随着时间的推移添加领域。保持这些智能体轻量和聚焦，以便它们响应更快，更不容易出错或产生幻觉。
* _基础化每个答案_。在手册、政策和发布说明上添加RAG工具。在提示指令中，使用“仅使用提供的上下文”这句话。但更进一步，使用基础化智能体来确保上下文答案始终是基础化的。
* _将HITL作为一项功能_。暴露一个`escalate_to_human`工具；路由复杂或愤怒的案例和大额退款。提供一种机制，允许客户就智能体解决问题的效果给出直接反馈。
* _身份和访问_。在暴露账户数据之前验证用户身份；对工具遵循最小权限原则。授予智能体用户的角色和关联工具以控制访问。
* _弹性_。实施重试、回退到次级模型/工具、超时和友好的失败消息。一个健壮的智能体向用户表明已采取措施解决他们的问题，增加他们的信心。
* _缓存和速率限制_。缓存热门答案（“退款政策”）并限制外部API以保护成本和可靠性。明智地使用缓存，避免“存储一切”的心态。
* _透明追踪_。保留每会话追踪以审计决策，包括执行的搜索和引用的来源。这可能涉及从Phoenix提取追踪或复制它们用于支持、审计和数据捕获目的。

清单11.5演示了一个简单的分类/升级智能体，它可以作为客户支持系统的第一个接触点。此智能体控制用户的问题或陈述是需要人工升级还是可以由检索智能体回答。

   return f"Escalated ticket {ticket_id}: {reason}"    #2

retrieval_agent = Agent(... )  #3

   name="Triage Support Agent",

       "Verify identity before account actions. Cite policies. "
       "If unsure or user is upset, call escalate_to_human."
       "Pass on complex queries to retrieval_agent."    #4

在标准的客户支持工作流中，在用户提出问题后，一个分类智能体确定它是可以解决还是需要传递给人类。如果它可以协助用户，流程接下来移交给一个检索智能体。此智能体提取与用户问题相关的信息，并生成评论、摘要或对原始问题的答案。然后此输出由一个基础化智能体检查，该智能体验证输出是否完全基于所提供的上下文。

## 11.3 构建RAG智能体系统的技巧

RAG智能体通常是其他智能体系统的骨干。正如我们在上一节中看到的，RAG智能体可以嵌入到更大的智能体系统中，例如客户支持智能体。我们很少看到单个RAG智能体直接向用户提供输出。

检索是引擎；智能体控制添加了路由、评论和围绕它的工具。但就像客户支持示例一样，一个RAG智能体系统由多个智能体组成，每个智能体在更大的RAG工作流中具有特定角色。使用以下技巧来指导此类智能体的开发：

* _仅在需要时使用智能体_。如果一次性RAG调用有效，则跳过编排器。为路由（分支检索）、诊断或行动添加智能体。专注于只保留对更大工作流至关重要的智能体。
* _设计模块化智能体_。分类（路由器）> 检索器 > 回答者；可选地，一个用于纠正性检索（CRAG）的批评者。
* _优化检索_。ANN索引、元数据过滤器、领域调整嵌入和重新排序。包括混合检索选项以找到各种搜索形式。
* _基础化纪律_。强制“仅使用上下文”并要求引用或来源摘录。包括专门的基础化智能体来评估答案上下文。
* _可观测性和评估_。记录检索召回率、答案准确性以及系统应该说“我不知道”的情况。跟踪检索调用可以帮助识别数据搜索过程中的痛点。在几乎所有情况下，你都希望你的智能体系统回复“我不知道”，而不是提供一个带有自信的错误答案。

清单11.6演示了一个用于检索和答案生成的单个智能体。在实践中，这样的智能体将嵌入到更大的智能体系统中。我们在前几章中已经研究了带有基础化智能体的RAG智能体模式。这里的示例提供了一个关于此类智能体所需重要技巧的简单提醒。

## 11.4 构建深度研究智能体系统的技巧

深度研究智能体混合了开放式的网络探索、迭代规划和来自许多来源的综合。几乎所有主要LLM提供商（ChatGPT、Claude和Gemini）都支持此类智能体。如果你需要一个深度研究智能体来搜索其他信息来源，如知识库、内部系统数据库和文档存储，我们通常建议构建一个内部版本。

构建一个内部深度研究智能体可能很复杂，并且需要开发一个专门智能体团队。仅当你确定更简单的智能体系统可能无法解决复杂问题时，才使用这种类型的智能体。以下是开发这些类型的智能体系统的一般最佳实践：

* _两层编排_ — 一个研究规划者（大脑）将任务委派给无状态的_工作者_（网络搜索器、提取器、分析器、总结器）。保持工作者像工具一样（没有长期记忆）。
* _工具政策控制_ — 强制规划者调用检索和网络工具获取事实。禁止在没有来源的情况下直接回答主张。
* _自我评论和事实检查_ — 在最终确定之前添加一个评论者传递以检查覆盖范围、矛盾和缺失视角。
* _流式用户体验_ — 流式传输部分发现和不断演变的大纲。这改善了感知延迟并支持“检查点”批准。这可以是一个静态的消息列表，如“正在搜索或思考”，或者是智能体正在输出的实际推理思想。
* _缓存和预算_ — 缓存重复查询并设置每次运行的令牌预算并带有警报。限制网络/API调用。当智能体重复相同查询时，这可能是一个很大的资产，但它也可能阻碍搜索。因此，确保定期清除缓存以避免返回陈旧数据。

## 总结

* 将智能体设计视为五个堆叠层：人格、工具与行动、推理与规划、知识与记忆，以及评估与反馈。独立优化每一层，然后集成。
* 将人格写成一个API契约，而不是散文：一个清晰的角色、边界和明确的“我不知道”退出路径。
* 保持范围狭窄（单一职责）以获得更高的准确性和更容易的测试/成本控制。
* 优先使用结构化、类型化的输出（Pydantic/数据类JSON）以消除解析脆弱性。
* 在运行时使用动态指令（注入日期/组织/用户）以避免过时事实。
* 构建具有清晰名称、文档字符串和类型参数的单一职责工具。模型从这些模式中学习工具使用。
* 控制工具何时运行（例如，`tool_choice`，并行工具调用）；当一个调用足够时提前停止。
* 在合理的情况下使用预构建或托管工具，但要有所选择。
* 为故障做计划：超时、重试、优雅的错误负载和回退。
* 对非平凡任务采用ReAct或思维链模式。
* 计划，然后执行：在行动前起草一个简短检查清单；考虑一个轻量级的“思考”服务器或步骤。
* 限制迭代以防止失控循环。在预算/轮次限制时发出`needs_followup`信号。
* 在最终确定前添加快速自我审查传递以捕获明显错误。
* 将RAG视为一等公民：使用知识库上的检索工具，而不是将文档塞入提示词。
* 将会话记忆（短期线程上下文）与长期记忆（向量/数据库）分开，并积极修剪。
* 优化检索：使用ANN索引，按领域分片，用元数据过滤，并考虑领域特定或多语言嵌入。
* 使用混合搜索（语义加关键词加层次/图）以提高召回率和精确度。
* 语义分块，严格将答案基础化在检索到的上下文中，并乐于说“不知道”。
* 保持数据新鲜并按角色/租户分区以强制访问边界。
* 追踪一切：提示词、工具调用、令牌计数、延迟和结果；将它们连接到你的可观测性栈。
* 为每次提示词/模型更改自动化评估（LLM作为评判者加策划的测试集）。
* 融入HITL：收集用户反馈，审查复杂案例。
* 在输入、工具调用和输出处添加护栏和审核；验证模式。
* 实践AIOps：测量 > 观察 > 迭代，包括你的智能体依赖的下游系统（索引/API）。
* 首先缩小职责范围（订单、退货、状态）；逐步扩展。
* 通过RAG基础化每个答案；如果需要，添加专门的基础化智能体。
* 提供HITL升级（`escalate_to_human`）并捕获用户反馈。
* 在暴露账户数据或调用业务API之前强制执行身份和最小权限访问。
* 设计弹性（重试、回退、超时）并缓存热门答案；应用速率限制。
* 维护每会话透明追踪以实现可审计性。
* 现实世界的“支持智能体” = 多个协作智能体（分类 > 检索 > 基础化 > 行动/护栏 > 答案）。
* 仅在需要时使用智能体编排器；如果一次性RAG足够，保持简单。
* 偏好模块化角色：路由器/分类 > 检索器 > 回答者；可选地，一个用于纠正性检索的评论者（CRAG）。
* 优化检索（ANN、过滤器、重新排序器、HyDE）并用引用强制基础化。
* 检测召回率、准确性和“应该说我不知道”的案例，并通过评估闭环。
* 对于高级系统，支持多个语料库、排名/细化循环和有限重试。
* 使用两层编排：一个规划者（大脑）将任务委派给无状态的工作者（手）进行网络搜索、提取、分析和写作。
* 强制执行工具使用政策：没有引用来源的主张；事实必须来自检索或文档工具。
* 在最终报告前添加自我评论/事实检查（评论者传递）。
* 提供流式用户体验以在出现时表面化部分发现和大纲。
* 管理缓存和预算：去重重复查询，限制令牌和调用，并定期清除缓存以避免过时。
* 生产就绪的智能体系统源于严格的人格、范围明确的工具、引导的推理、纪律严明的检索和不断的测量。这些元素被组装成模块化的、可审计的工作流，具有清晰的升级路径和基础化的输出。
